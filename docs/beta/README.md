# `docs/beta/` — Phase 6.0 controlled-beta pack

This directory holds the `oida-code` Phase 6.0 controlled-beta
operator pack. It is the entry point for an external operator who
has been invited to the closed beta.

> Phase 6.0 is a controlled beta. It is **not** a public launch.
> The pack here is for invited operators — please do not share
> outside the invite list.

## Where the metric script lives

The aggregator that turns submitted feedback forms into a single
`beta_feedback_aggregate.md` is at:

* **[`scripts/run_beta_feedback_eval.py`](../../scripts/run_beta_feedback_eval.py)**

It tolerates the zero-feedback case cleanly (when no operator has

## Where beta cases and feedback land

Beta cases and submitted feedback live under the `reports/beta/`
directory:

* `reports/beta/beta_cases.md` — the running registry of beta
  cases (one row per case).
* `reports/beta/beta_feedback_aggregate.md` — generated by the
  aggregator script.
* `reports/beta/beta_case_<n>_<short_label>.md` — one file per
  case, copied from [`beta_case_template.md`](beta_case_template.md).
* `reports/beta/beta_case_<n>/beta_feedback_<run_id>.yaml` — one
  feedback form per completed run.
* `reports/beta/beta_case_<n>/bundle/` — the gateway-bundle for the
  case (operator-authored).

## What stays the same in beta

The Phase 6.0 controlled beta does **not** change the structural
defaults of the gateway path:

* `enable-tool-gateway` stays **default false** in the composite
  Action input. Operators set it `true` per workflow run, never
  project-wide. The recommendation key in
  `beta_feedback_aggregate.json` is diagnostic only and never
  flips this default.
* Official fields (`total_v_net`, `debt_final`, `corrupt_success`,
  `corrupt_success_ratio`, `verdict`) stay **structurally
  blocked** — pinned as null / not-emitted across every
  artefact, every schema, and every output. The block is not
  relaxed in beta.
* `gateway_status` on every report stays `diagnostic_only`. No
  beta artefact ever produces a product verdict.

## What the beta cannot tell you

(One last reminder — repeated from `beta_known_limits.md`.)

* It cannot tell you a PR is safe to merge.
* It cannot tell you a codebase is bug-free.
* It cannot tell you a security review is complete.
* It cannot tell you the LLM judgement is correct.
* It cannot tell you the gateway path generalises to your full
  codebase. The beta is **scoped to a named claim with a named
  pytest target**.
* It cannot tell you predictive performance numbers.

## Honesty statement (Phase 6.0 frame)

Phase 6.0 runs a controlled beta of the opt-in gateway-grounded
path with selected operators and controlled repos. It does not
make the gateway default. It does not implement MCP. It does not
enable provider tool-calling. It does not validate production
predictive performance. It does not emit official `total_v_net`,
`debt_final`, or `corrupt_success`. It does not modify the
vendored OIDA core.

## Questions

If a step is unclear, do not ask an LLM. Ask via the tracking
issue you were invited to, or ask the project author directly.
The whole point of the beta is to collect human signal on
exactly the steps that are unclear.
