## One Sentence

`oida-code` is a diagnostic second opinion for Python reviewers who
need to know whether claims about an AI-authored diff are supported,
contradicted, or unsupported by executable evidence.

It is not a merge gate and it does not certify code as ready for
production.

## Primary User

The primary user for the next 30 days is a Python maintainer, tech
lead, or CI reviewer who receives diffs produced or modified by coding
agents.

Secondary users, including OIDA researchers, platform teams, and future
SaaS/GitHub App operators, are out of scope for product prioritization
until the primary user path is understandable and useful.

## Job To Be Done

When a change generated by an AI agent appears correct, help the
reviewer answer four questions quickly:

1. What exact claim is being made about the diff?
2. Which tool output supports that claim?
3. Which tool output contradicts or weakens it?
4. What still needs human inspection?

The output must be evidence-linked. It must not imply a final product
verdict.

## North Star Outcome

Within 10 minutes, an external Python reviewer can run or read the
diagnostic path, identify at least one load-bearing claim as supported,
contradicted, or unsupported, and understand that the report is not a
merge or production decision.

## Active Product Path

The active product path is:

1. A local CLI diagnostic report for a Python repo/diff.
2. A GitHub composite Action that publishes diagnostic artifacts.
3. An opt-in gateway-grounded diagnostic path, still default-false.
4. A small human-readability validation loop over the diagnostic report.

Everything else is research or backlog until this path is clear.

## Vocabulary

Use these terms for current product output:

- `supported_claim`: the named claim has relevant executable evidence.
- `contradicted_claim`: the named claim conflicts with tool output.
- `unsupported_claim`: the run did not establish support or contradiction.
- `diagnostic_only`: the report helps a reviewer inspect evidence.

Avoid turning research concepts into product promises. In particular,
`verification_candidate` is an internal diagnostic status, not a user
promise that the code is correct.

## G-6d Position

G-6d remains scientifically important. The live corpus after ADR-73 is
still N=14 (10 train, 4 holdout), and the target remains N>=20.

However, G-6d is not the immediate product strategy. Any next G-6d
pinning block must inherit the recorded ADR-75 dependency policy and ADR-76
exact-four stop rule before freeze.

ADR-75 records the dependency policy: reject or defer such candidates
before freeze for G-6d.4. ADR-76 records the matching tranche policy:
G-6d.4-style pinning requires an exact clean +4 split +3 train / +1 holdout;
fewer clean candidates means stop before freeze, not partial freeze. Do not
add requirements-file install support as a post-freeze rescue for a failing
case. Any future support for requirements-file / tox-only dependency patterns
requires a separate ADR before candidate selection and a structural update to
the predeclared clone-helper flag tests.

## Phase 6.e Front Door

Phase 6.e aligns the existing `audit --format markdown` path with this product
strategy. The human Markdown report is now a diagnostic reviewer surface: it
does not present internal legacy JSON verdict labels as product claims, does
not add a new alias, and does not change JSON/SARIF compatibility.

## Phase 6.f GitHub Action Metadata

Phase 6.f aligns the reusable `action.yml` public metadata with the same
diagnostic-only product strategy. The action description now presents the
Action as a diagnostic artifact producer for AI-authored Python diffs, not an
AI code verifier, merge gate, production-readiness decision, or guarantee of
behavior. Action defaults and runtime behavior are unchanged.
