# Project Brief — oida-code

## Purpose

Ship a CLI-first **AI code verifier** that measures the gap between what AI-written code *appears* to do and what it *actually* guarantees.

The product does NOT claim "mathematical proof" of arbitrary code semantics (Rice's theorem forbids it). It emits one of four verdict buckets per change set:

1. **proved enough for merge** — formal proof of an explicit property OR behavioral evidence above thresholds.
2. **counterexample found** — execution produced a failing case.
3. **insufficient evidence** — cannot confirm nor refute within budget.
4. **high apparent quality / negative net value** — *corrupt success*: `Q_obs` high, `V_net < 0`. This is the wedge.

Grounded in the OIDA v4.2 formal model (Abadie, 2026) — a trace-based stock-flow model of operational debt in tool-using AI agents.

## Target Users

- **Individual developers** running `oida-code audit` against PRs generated by coding agents (Cursor, Claude Code, Copilot, aider, etc.), before merging.
- **Teams** integrating `oida-code` as a GitHub PR check (phase 2+).
- **Platform engineers** auditing AI-driven CloudOps changes (phase 3+).

## Constraints

- **Languages (v0):** Python only. Blueprint §4.
- **Honesty:** blueprint §12 rules are non-negotiable. No "mathematical proof" claims outside the 4 buckets.
- **Reuse:** the existing OIDA deterministic scorer (`search/OIDA/oida_framework/`) is vendored verbatim under `src/oida_code/_vendor/oida_framework/`. The blueprint forbids re-implementing the grounding / Q_obs / μ / λ_bias / V_net formulas.
- **Local inference:** default LLM verifier is Qwen3.6-35B-A3B Q4_K_M via llama.cpp on an RTX 3500 Ada Laptop (12 GB VRAM); smaller local fallback for classification.
- **Hardware floor:** the dev laptop has 80 GB free on C: — any local LLM work is blocked until storage is upgraded (M.2 2 TB).
