# Critique by `oida-code` adversarial lane

## Summary
The beta operator pack assumes too much internal context and fails to explain the most essential prerequisite: what each of the eight bundle files must contain and how to author them. A cold reader cannot proceed past “author a bundle” without reverse-engineering examples or seeking project-author intervention, which defeats the controlled-beta goal.

## Confusion points (jargon, undefined terms)
- `beta_operator_quickstart.md`: “If you have not used a "verifier loop" before: the closest analogue is "a property-based test plus a structured argument tracing the test back to a named property".” — The analogue still doesn’t explain what a verifier loop actually *does* in this system (e.g. how passes interact), leaving a cold reader guessing.
- `beta_known_limits.md`: “Authoring it by hand is non-trivial and the beta is the first external test of "is this feasible".” — The phrase “non-trivial” is vague; it could mean “hard” or “the format is undocumented”, and the pack never clarifies which, which causes confusion about whether the reader is missing instructions.
- `beta_operator_quickstart.md`: The CLI launch example uses `--pass1-forward`, `--pass1-backward`, `--pass2-forward`, `--pass2-backward` but never explains what each pass does, why they exist, or what they contain. Without that, the reader cannot map the bundle files to anything meaningful.
- `beta_case_template.md`: “**Claim type:** <<<one of: capability_sufficient | benefit_aligned | … | shadow_pressure_explained>>>” — `shadow_pressure_explained` appears nowhere else in the provided pack and is never defined; a cold reader has no idea what it means or when to use it.
- `beta_feedback_form.md`: “evidence_traceability … each claim has a clear, single-hop link to the evidence item that supports it (e.g. `[E.event.1] → tool: pytest, scope: tests/test_x.py, summary line: "12 passed in 0.3s"`).” — The `E.event.1` identifier is introduced as an example, but the pack never explains how an operator creates evidence items in the bundle or how those identifiers are assigned.
- `beta_operator_quickstart.md`: “The runners reject any response that contains any of these phrases.” — “Runners” is used as a load-bearing term without definition; a new reader might not know it means the pipeline components that scan for forbidden text, not the GitHub Actions runner.

## Contradictions / inconsistencies
- `beta_known_limits.md`: “Authoring it by hand is non-trivial and the beta is the first external test of "is this feasible".” vs `beta_operator_quickstart.md`: “Authoring the bundle takes ~30–60 minutes for an experienced Python developer reading the runbook for the first time.” — The known-limits document treats feasibility as an open question, while the quickstart gives a firm, low time estimate that implies feasibility is already demonstrated; this dissonance makes the reader uncertain about whether the bundle authoring is genuinely doable.
- `README.md`: “The aggregator that turns submitted feedback forms into a single `beta_feedback_aggregate.md` is at: `scripts/run_beta_feedback_eval.py`” and “It tolerates the zero-feedback case cleanly” while `beta_feedback_form.md` says: “The aggregator does NOT silently default missing fields — it rejects the form with a clear error so the operator can fix it.” — These are not contradictory (one is about missing feedback, the other about missing fields), but the juxtaposition can confuse a cold reader about what the script rejects vs tolerates.

## Verdict-leak risk
- `docs/concepts/oida_code_plain_language.md`: “`verification_candidate: true` is the **strongest positive signal** the project emits.” — The bolded phrase invites a product-like reading (“strongest positive” implies relative safety/reliability) despite the immediately following disclaimers. A hurried cold reader could easily land on “strongest positive signal” and conclude the claim is good enough to merge, leaking a pseudo-verdict.
- `beta_operator_quickstart.md`: “If the gateway status is anything other than `diagnostic_only`, that is a project bug. Stop and report it.” — The surrounding framing that the operator must treat non-diagnostic status as a bug could be misread as the project guaranteeing that the diagnostic *never* implies a verdict, which is correct, but the emphasis on “stop” may make the reader think the project claims the block is absolute and therefore the artifact is “safe” to rely on, when the policy forbids such a meta-claim. This is a weaker leak, but worth noting.

## Bundle authoring blockers
- **No specification of the eight bundle files**: `beta_case_template.md` says “Author the bundle under `reports/beta/beta_case_<n>/bundle/`” but never describes what `packet.json`, `pass1_forward.json`, `pass1_backward.json`, `pass2_forward.json`, `pass2_backward.json`, `tool_policy.json`, `gateway_definitions.json`, or `approved_tools.json` must contain. The closest guidance is the quickstart’s smoke-test command that references the keystone example, but that only shows the example, not how to derive your own.
- **No explanation of the allowed fields or claim structure**: The operator is told to pick a claim type from a list, but the pack gives no example of a concrete claim string, how to phrase it, or which surface/field corresponds to the claim. The `beta_case_template.md` placeholder `<<<C.<surface>.<claim>>>>` assumes internal knowledge of the `LLMEvidencePacket` schema.
- **The quickstart’s bundle-authoring instruction is circular**: “Author a bundle (use [`beta_case_template.md`](beta_case_template.md)).” — The template only contains a section heading “Bundle authoring notes” for prose, not a how-to, so the user is directed to a file that doesn’t help them author the bundle.
- **No accessible example of a custom bundle apart from the keystone self-audit**: The quickstart smoke test uses the shipped example that targets `oida-code` itself; that example is not explained, so the operator must treat it as a cryptic template and reverse-engineer it, which is disproportionate to the beta’s promise of a 10-minute quickstart.
