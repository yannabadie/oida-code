# Methodology critique by <PROVIDER>/<MODEL>

## Summary

The Phase 6.1' chain demonstrates rigorous discipline documentation but exhibits a critical asymmetry: the headline "partial holdout generalisation" (1/2) is presented as a success while the 1/2 failure is attributed to a seed-record authoring quality issue that the discipline itself should have caught earlier. ## Verdict-leak risk

- `reports/phase6_1_h_freeze_pass.md`: "This is the FIRST holdout-target outcome where the verifier accepts a claim through the entire grounded path." — While this does not use a forbidden phrase, the framing "FIRST holdout-target outcome" combined with "accepts a claim" could be misread by a downstream consumer as "the tool works on holdout data" — a generalisation claim that the 1/2 success rate does not support. The phrase "accepts a claim" is verifier-internal language, but in a summary document it reads like a product endorsement.

- `reports/phase6_1_h_freeze_pass.md`: "The chain CAN now honestly claim **partial holdout generalisation**" — "Partial holdout generalisation" is not a forbidden phrase, but it is a verdict-like claim about the system's capability. The no-product-verdict policy forbids "merge-safe", "production-safe", etc., but "generalisation" is a softer verdict that implies the system has a property (generality) that would be relevant to production use. A reader could reasonably interpret this as "the system partially works on unseen data."

- `reports/phase6_1_e_steps_1_3.md`: "This is the **first claim-supporting outcome** in the calibration_seed lane." — "Claim-supporting outcome" is verifier terminology, but in a phase report it reads like a milestone announcement. Combined with the bold formatting, it risks being read as "the tool finally works" rather than "a specific configuration produced a specific internal state."

## Discipline / Goodhart risks

- **Ratio guard at N=5 is too tight to be meaningful.** The holdout ratio guard [0.20, 0.40] is enforced at N_pinned=5, meaning the guard activates when there are exactly 5 pinned cases. At N=5, a single case moving from train to holdout changes the ratio by 0.20 (the entire allowed band). The guard is technically active but statistically meaningless — the difference between 1 holdout (0.20) and 2 holdouts (0.40) is a single case, not a meaningful split. The discipline should acknowledge that the ratio guard is a formality at N<20, not a real statistical safeguard.

- **The freeze-rule carve-out for "predeclared env bootstrap" is dangerously broad.** The freeze rule explicitly allows `--scm-pretend-version`, `--install-extras`, `--install-group`, and `--import-smoke` as "predeclared env bootstrap" (per cgpro verdict_q1). But these flags directly affect whether the holdout pass succeeds or fails. If the freeze rule allows adding flags that make holdouts pass, the freeze rule is selectively permeable. The distinction between "env bootstrap" and "tooling edit" is not operationally defined — adding `--install-extras test` is a tooling edit to the clone helper that was made specifically because holdouts failed. The carve-out should be explicitly bounded: only flags that existed BEFORE the holdout pass was designed, not flags added in response to holdout failures.

- **Seed authoring quality defects on seed_157 imply more undiscovered defects.** The Phase 6.1'h report admits that seed_157's test_scope was "over-broad" — the operator-authored Tier-3 field was wrong. But only 3 of 46 seed records are Tier-3-complete and pinned. If the operator's Tier-3 authoring quality is imperfect on a pinned case that was manually reviewed, the 43 unpinned records likely contain similar or worse defects. The discipline has no mechanism to audit Tier-3 authoring quality across the corpus.

- **The "observation-only investigation" carve-out is undefined.** Phase 6.1'h says "Observation-only investigation of seed_157 root cause (running specific tests separately, no tooling edit) ✅ allowed." But "observation-only" is not defined in any ADR or schema. The operator ran specific pytest commands to diagnose the failure — this is investigation, not observation. If the freeze rule allows arbitrary investigation as long as no files are edited, the rule is unenforceable because the investigator's knowledge changes regardless of file edits.

## What would make you doubt the chain's "partial generalisation" claim

1. **The generalisation claim rests on a single holdout success (seed_065) that was achieved only after two dedicated corrective phases (6.1'f and 6.1'g) motivated by the holdout failure.** If the bootstrap fix was target-class-general, why did it require two attempts? The first fix (install-order flip) was insufficient; the second fix (extras + groups) was needed. This suggests the bootstrap is not truly general — it was iterated until it worked on the specific holdout cases.

2. **The LLM-authored replays are unaudited.** Both `verification_candidate` outcomes depend on DeepSeek-authored replay files that are not independently verified. If the LLM hallucinated test outputs that happen to match the verifier's acceptance criteria, the `verification_candidate` is meaningless. The methodology should include a replay-audit step (e.g., human review of a random sample of LLM-authored replays) before claiming generalisation.

3. **The seed_157 failure is attributed to seed-record quality, but the same operator authored seed_065's Tier-3 fields.** If the operator's Tier-3 authoring is unreliable on one holdout, why should we trust it on the other? The discipline has no mechanism to distinguish "good authoring" from "bad authoring" except post-hoc rationalisation. The generalisation claim would be stronger if the operator had authored multiple holdout cases and all succeeded, or if there was an independent audit of Tier-3 field quality.

## Honest uncertainty

1. **Is the "predeclared env bootstrap" carve-out consistent with the freeze rule's intent?** The freeze rule is supposed to prevent the generator from being tuned against holdout cases. But the bootstrap fixes (install-order flip, extras, groups) were made specifically because holdouts failed. The carve-out says these are "env bootstrap, not tooling edits" — but I am uncertain whether this distinction is meaningful or whether it creates a loophole that makes the freeze rule selectively permeable. The chain's own documents acknowledge this tension (Phase 6.1'f says "fixing it in the same block would blur whether the minimal hypothesis closed the observed failure class") but never resolve it.

2. **Would a different seed_157 test_scope have produced `verification_candidate`?** The Phase 6.1'h report shows that the two specific tests pass individually. If the operator had authored `test_scope` as the two specific test names instead of the class name, seed_157 might have been a second `verification_candidate`. This would change the generalisation tally from 1/2 to 2/2. I am uncertain whether the "honest negative" classification is robust or whether it is an artifact of a single operator's authoring choice that could have gone either way.

3. **Is the corpus selection bias (maintainer-authored, non-fork PRs) actually a problem for the methodology?** The chain repeatedly flags the fork-PR fence as a selection-effect caveat, but never demonstrates that the bias matters. The holdout cases are maintainer-authored (sqlite-utils by Simon Willison, structlog by Hynek Schlawack), and the pipeline works on one of them. Would the pipeline also work on community-contributed PRs? The methodology cannot answer this because the fork-PR fence prevents collecting such cases. I am uncertain whether this is a real limitation or a self-imposed constraint that the chain has accepted.
