forward verifier:

backward verifier:

  accepte seulement les claims supportés par les deux directions

Règle d’agrégation

Le claim n’est accepté que si :

1. forward says supported
2. backward says necessary conditions met
3. all evidence_refs exist in packet
4. no deterministic tool contradiction
5. claim_type is allowed
6. confidence within cap
7. claim does not mention forbidden official fields

    ...

backward-only support = insufficient

Forward verifier contract

Le forward verifier vérifie la suffisance :

Étant donné ces preuves, quelles conclusions peut-on soutenir ?

Exemple :

Evidence:
  [E.prec.1] guard_detected verified
  [E.test.1] negative_path_tested unverified
  [E.graph.1] precondition -> api_contract constitutive

Forward result:
  accepted:
    precondition_supported
  rejected:
    negative_path_covered
    observability_sufficient

Le forward verifier ne peut pas inventer une conclusion qui n’est pas reliée à une evidence ref.

Backward verifier contract

Le backward verifier vérifie la nécessité :

Pour soutenir cette conclusion, quelles preuves seraient nécessaires ?
Sont-elles présentes ?

Exemple :

Claim:
  observability_sufficient

Required:
  at least one test_result OR logging/error-surface evidence
  negative path evidence if failure mode is user-visible

If absent:
  necessary_conditions_met = False

C’est cette couche qui empêche le LLM de dire “observability high” juste parce qu’il voit un test global vert.
