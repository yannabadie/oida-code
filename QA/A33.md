Décision Phase 5.6

Phase 5.6 doit exposer le gateway dans l’action GitHub, mais seulement comme chemin expérimental opt-in.

Objectif :

action.yml
  enable-tool-gateway=false par défaut

si enable-tool-gateway=true:
  exécuter verify-grounded sur un bundle replay explicite
  produire des artefacts gateway
  afficher un résumé diagnostic-only

Non-objectifs :

- ne pas rendre verify-grounded par défaut
- ne pas exécuter le gateway sur PR fork
- ne pas appeler provider externe
- ne pas ajouter MCP
- ne pas activer provider tool-calling
- ne pas exécuter d’outils write/network
- ne pas émettre total_v_net / debt_final / corrupt_success
- ne pas dire merge-safe / production-safe / bug-free

Point sécurité important : même si le gateway est local et déterministe, certains outils comme pytest peuvent exécuter du code du dépôt. Le chemin gateway doit donc être interdit sur code non fiable, notamment PR fork. GitHub documente que permissions permet de réduire les droits du GITHUB_TOKEN, que pull_request_target accorde un token read/write même depuis un fork public, et recommande de limiter les permissions au minimum requis.

5.6-A — inputs action.yml

enable-tool-gateway existe déjà comme réservé. Phase 5.6 doit l’implémenter.

Inputs proposés :

enable-tool-gateway:
  default: "false"
  description: "Experimental. Run gateway-grounded verifier with an explicit replay bundle."

gateway-bundle-dir:
  default: ""
  description: "Directory containing packet.json, replay files, tool policy, gateway definitions and approved tools."

gateway-output-dir:
  default: ".oida/gateway-grounded"
  description: "Where gateway-grounded verifier artifacts are written."

gateway-fail-on-contract:
  default: "false"
  description: "Fail only on contract violations, never on diagnostic findings."

Règles :

enable-tool-gateway=false:
  comportement Phase 4.9 inchangé

enable-tool-gateway=true:
  gateway-bundle-dir obligatoire
  bundle validé avant exécution
  exécution interdite sur pull_request / fork PR
  replay/fake only
  no external provider
  no MCP
  no write/network

GitHub explique que les composite actions servent à regrouper des étapes réutilisables, mais rappelle aussi que certains contextes peuvent contenir de l’input non fiable. C’est précisément pour ça que les inputs du bundle doivent être passés par env: puis utilisés comme variables shell, jamais interpolés inline dans un run:.
