QA/A41.md — Phase 6.0: controlled beta
Décision

Phase 6.0 doit être une beta contrôlée sur un petit nombre d’utilisateurs/repos, pas une nouvelle phase d’architecture.

Objectif :

Vérifier qu’un opérateur externe peut utiliser le chemin gateway opt-in,
comprendre les artefacts,
prendre une décision humaine utile,
et remonter les points de friction.

Non-objectifs :

- ne pas activer enable-tool-gateway par défaut
- ne pas publier une promesse production
- ne pas créer GitHub App
- ne pas utiliser Checks API custom
- ne pas ajouter MCP
- ne pas activer provider tool-calling
- ne pas exposer total_v_net / debt_final / corrupt_success
- ne pas dire merge-safe / production-safe / bug-free

NIST présente TEVV — test, evaluation, validation and verification — comme un ensemble d’activités pour concevoir et conduire des évaluations significatives, avec tâches, testbeds, outils et données caractérisées ; Phase 6.0 doit rester dans cette logique : mesurer un usage contrôlé, pas proclamer une validité produit.

Pourquoi Phase 6.0 maintenant

On a trois signaux suffisants pour passer en beta contrôlée :

Le pipeline technique est stabilisé.
Le chemin gateway opt-in, les artefacts, le Step Summary, les rapports Markdown, le SARIF, les outputs d’action et les manifests sont documentés et testés.
Le soak opérateur a produit 5 cas utiles.
Les 5 cas sont complétés, tous useful_true_positive, avec operator_usefulness_rate = 1.0, UX moyenne 2/2/2/2, et zéro fuite de champ officiel.
La documentation est maintenant utilisable.
Le guide explique quand utiliser le gateway, quand ne pas l’utiliser, comment préparer un bundle, comment lancer un run, et comment lire les artefacts.

Mais on n’a pas encore :

- usage par quelqu’un hors Yann / Claude / cgpro
- diversité de workflows humains
- retour qualitatif externe
- mesure de friction d’installation
- mesure du temps de compréhension
- vrais faux positifs / faux négatifs hors cas contrôlés

Donc la bonne suite est une beta fermée.
