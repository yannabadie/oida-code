ADR-21 — Minimal dependency graph before graph-aware fusion

Decision:
  Block C builds a bounded dependency graph for repair propagation and audit planning.
  It does not claim graph-aware V_net, because the vendored OIDA analyzer currently
  does not use constitutive/supportive edges inside the V_net computation.

Definitions:
  constitutive edge A -> B:
    B is not valid if A is invalid.

  supportive edge A -> B:
    A should be audited when B is suspicious, or B should be audited when A changes,
    but B is not automatically invalid.

Rejected:
  - full transitive call graph
  - graph edges without evidence
  - using graph presence to unlock official V_net/debt

Règles d’arêtes
1. Même scope / même fonction

Si deux obligations partagent le même fichier et le même symbole, ou si une obligation est plus basse niveau qu’une autre dans le même scope :

precondition -> api_contract = constitutive
invariant -> api_contract = constitutive
security_rule -> api_contract = constitutive
migration -> api_contract = supportive sauf preuve directe data-path
observability -> api_contract = supportive

Exemple :

src/app.py::create_user precondition
    -> src/app.py::create_user api_contract

Raison :

same_symbol_guard_supports_contract
2. Imports directs

Utilise l’AST Python pour détecter Import et ImportFrom. Le module ast est le bon outil ici : la doc officielle indique que ast.parse() produit l’arbre syntaxique, que les imports sont des nœuds Import / ImportFrom, et que NodeVisitor sert précisément à parcourir l’arbre.

Règle :

imported module obligation -> importing module obligation = supportive

Pas constitutive par défaut.

Pourquoi : un import peut être nécessaire, mais en Block C on ne sait pas encore si le symbole importé est réellement celui qui fonde le contrat. On évite donc les faux “reopen”.

Exemple :

src/db.py obligation
    -> src/service.py obligation
kind = supportive
reason = direct_import
3. Tests liés

Si un fichier de test est lié par convention :

tests/test_foo.py -> src/foo.py
src/foo.py -> tests/test_foo.py

Mais dans les parents OIDA, je préfère :

test obligation -> source obligation = supportive

Pas constitutive.

Un test manquant ou cassé doit déclencher un audit, pas prouver automatiquement que le code source est invalide.

4. Config / migration

Règles minimales :

pyproject.toml / setup.cfg / tox.ini / pytest.ini -> all changed Python obligations = supportive
migration file -> data/API obligations in same changed set = supportive
migration file -> migration obligation itself = constitutive only within same migration scope
5. Pas d’arêtes inventées

Chaque arête doit avoir :

reason
source
confidence

Et chaque parent_id / child_id doit exister. Le core OIDA valide déjà les IDs et lève une erreur si un parent inconnu est référencé, donc il faut garder cette garantie forte.

Borne stricte

Ne fais pas de graphe global.

max_depth = 1
max_files = 50
external imports ignored
stdlib ignored
site-packages ignored
unresolved imports recorded, not guessed

On pourra passer à max_depth=2 plus tard, mais pas maintenant.

Direction des edges

Très important :

A -> B signifie :
B dépend de A.

Donc si src/service.py importe src/db.py :

db_event -> service_event

Pas l’inverse.

Pourquoi : dans double_loop_repair(root_event_id), invalider le root doit rouvrir ou auditer ses descendants. Le core utilise un DiGraph, puis des dominators pour calculer les descendants à rouvrir ou auditer ; NetworkX définit bien DiGraph comme un graphe à arêtes dirigées, et immediate_dominators() travaille sur un graphe dirigé depuis un nœud de départ.
