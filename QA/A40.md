J’ai vérifié le repo. Mon verdict est clair :

> **Le projet a maintenant passé le cap “preuve technique interne”.
> La bonne direction n’est plus d’ajouter de nouvelles briques, mais de transformer le chemin opt-in en usage opérateur documenté, reproductible et propre.**

## Où on en est

Le dernier rapport `phase5_8_x_pytest_summary_line.md` indique que le **Tier 5 operator soak gate est cleared** : cinq cas opérateur sont complétés, tous labellisés `useful_true_positive`, avec `official_field_leak_count = 0`, `operator_usefulness_rate = 1.0`, et la recommandation est passée à `document_opt_in_path`. Le rapport précise aussi que même avec cette recommandation, `enable-tool-gateway` reste `false` par défaut. ([GitHub][1])

L’agrégat opérateur confirme les cinq cas complétés : `case_001_oida_code_self`, `case_002_python_semver`, `case_003_markupsafe`, `case_004_python_slugify`, `case_005_voluptuous`, tous avec le label `useful_true_positive`, zéro faux positif, zéro faux négatif, zéro fuite de champ officiel, et une recommandation `document_opt_in_path`. ([GitHub][2])

Donc le gateway-grounded verifier a maintenant une base empirique minimale : il a tourné sur plusieurs repos, plusieurs formes de claims, plusieurs stratégies d’installation, avec des labels humains/cgpro et des artefacts exploitables.

## Ce que ça veut dire

Tu peux maintenant **documenter l’usage opt-in** du gateway.

Tu ne dois pas encore :

```text
- le mettre par défaut ;
- dire que le projet est production-ready ;
- dire que le code est safe ;
- exposer total_v_net / debt_final / corrupt_success ;
- lancer MCP ;
- activer provider tool-calling.
```

Le signal positif est réel, mais il reste encore petit. Cinq cas utiles, c’est assez pour documenter un usage expérimental. Ce n’est pas assez pour en faire une promesse produit générale.

La prudence reste justifiée : le rapport Phase 3 avait déjà montré qu’un signal apparemment bon pouvait être confondu par la longueur de session et des proxies faibles.  Et la consultation v0.4.1 posait déjà le verrou dur : ne pas exposer `V_net`, `debt_final` ou `corrupt_success` tant que les champs porteurs et la validation ne sont pas solides. 

## Les problèmes restants

### 1. La documentation est probablement désynchronisée

J’ai vu une incohérence concrète : `reports/operator_soak/aggregate.md` dit que `case_002_python_semver` est complet et labellisé `useful_true_positive`, mais le README de `case_002` sur `main` dit encore que le cas est `awaiting_real_audit_packet_decision` et que le bundle n’est pas réel. ([GitHub][2])

Avant de documenter l’usage opt-in, il faut corriger ce type de drift. Sinon un lecteur externe va tomber sur deux vérités contradictoires.

### 2. Le chemin est encore trop manuel

Aujourd’hui, pour utiliser le gateway proprement, il faut comprendre :

```text
bundle
packet
pass1/pass2 forward/backward
tool_policy
gateway_definitions
approved_tools
operator labels
artefacts
summary
audit logs
```

C’est puissant, mais encore trop “framework interne”. La prochaine étape doit rendre ça opérable.

### 3. Les cinq cas sont positifs, mais pas assez adversariaux

Cinq `useful_true_positive`, c’est encourageant, mais il manque encore des cas :

```text
- false positive contrôlé ;
- false negative contrôlé ;
- tool timeout ;
- tool missing ;
- dépendance cassée ;
- test flaky ;
- output hostile ;
- fork PR bloqué en condition réelle.
```

Ce n’est pas bloquant pour documenter l’opt-in. Mais c’est bloquant pour toute prétention plus forte.

### 4. La décision `document_opt_in_path` ne doit pas devenir une promotion implicite

Le rapport est très clair : `document_opt_in_path` ne flippe pas `enable-tool-gateway`. Il reste `false`. ([GitHub][1])
C’est la bonne ligne.

Avec :

Le README doit permettre à quelqu’un de lancer :

et de comprendre le résultat.

### 4. Ajouter une page “interpretation guide”

Créer :

```text
docs/interpreting_gateway_reports.md
```

À expliquer :

```text
accepted claim ≠ code safe
diagnostic_only ≠ failure
verification_candidate ≠ official verification
pytest ok ≠ full behavioral proof
unsupported ≠ bug
blocked ≠ code failure
```

C’est crucial pour éviter les abus.

### 5. Ajouter une policy “no product verdict”

Créer ou renforcer :

```text
docs/security/no_product_verdict_policy.md
```

Rappeler que les termes suivants restent interdits :

```text
merge-safe
production-safe
bug-free
security-verified
official total_v_net
official debt_final
official corrupt_success
```

### 6. Ajouter un runbook opérateur public

Créer :

```text
docs/operator_soak_runbook.md
```

Plus simple que les fichiers internes `operator_soak_cases`.

Objectif : permettre à un opérateur de refaire un cas contrôlé.

### 7. Garder les tests de garde-fous

Phase 5.9 doit ajouter des tests surtout sur la doc :

```text
- aucun document ne dit enable-tool-gateway=true par défaut ;
- aucun document ne contient un faux verdict produit ;
```
