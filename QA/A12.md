E1.1 — point 1 : authoritative: Literal[False] n’est pas suffisant

Tu écris que authoritative: Literal[False] rend la mutation impossible. Ce n’est pas vrai en runtime Python si le modèle Pydantic reste mutable.

Le code actuel met seulement :

model_config = ConfigDict(extra="forbid")
authoritative: Literal[False] = False

dans ShadowFusionReport.

Or la documentation Pydantic indique que, par défaut, les modèles sont validés à la création, mais les changements d’attributs après création ne sont pas revalidés, sauf avec validate_assignment=True; et les modèles restent mutables sauf configuration explicite.

Donc ce claim :

authoritative: Literal[False] pinned by type — impossible to mutate to True

est trop fort.

Patch attendu :

model_config = ConfigDict(
    extra="forbid",
    frozen=True,
)

sur :

ShadowEventScore
ShadowGraphSummary
ShadowTrajectorySummary
ShadowFusionReport

Et idéalement remplacer les listes par des tuples :

event_scores: tuple[ShadowEventScore, ...] = ()
blockers: tuple[str, ...] = ()
warnings: tuple[str, ...] = ()

Pourquoi : un modèle frozen avec list[...] protège l’attribut, mais pas forcément la mutation interne de la liste. Un tuple rend le rapport vraiment plus sûr comme objet de sortie.

Tests E1.1 attendus :

test_shadow_report_authoritative_cannot_be_assigned_true
test_shadow_report_event_scores_cannot_be_appended
test_shadow_report_model_validate_rejects_authoritative_true
test_shadow_nested_scores_are_frozen

Le but n’est pas la sécurité absolue contre object.__setattr__, mais d’éviter qu’un intégrateur normal puisse muter le rapport et croire qu’il est devenu authoritative.

E1.1 — point 2 : contradiction entre “missing grounding” et shadow formula

E1.0 corrige bien la readiness :
si des préconditions existent mais que zéro est vérifiée, c’est un vrai signal négatif ; si aucune précondition n’existe, c’est missing, pas “zéro vérifié”. Le rapport E1 le formule correctement.

Mais dans experimental_shadow_fusion.py, _grounding_for(event) retourne encore :

if not event.preconditions:
    return 0.0

Donc la formule shadow traite un événement sans modèle de préconditions comme si son grounding réel était zéro.

Ce n’est pas bloquant pour E1 parce que la shadow fusion est non-authoritative, mais c’est exactement le genre de bruit qu’E2 doit corriger avant d’évaluer “keep/revise/reject”.

Décision recommandée :

real zero grounding:
  preconditions exist
  verified weight = 0
  => pressure high

missing grounding:
  no preconditions emitted
  => pressure neutral or component excluded
  => warning emitted

Implémentation simple :

def _grounding_pressure(event: NormalizedEvent) -> tuple[float, bool]:
    if not event.preconditions:
        return 0.5, False  # neutral, not real
    return 1.0 - _grounding_for(event), True

Ou mieux, mais plus lourd :

dynamic weighted mean:
  only include components whose source is real or heuristic
  if component is missing, exclude and renormalize weights

Pour E1.1, je recommande la version simple : neutral 0.5 + warning.
La renormalisation dynamique pourra être testée en E2 comme variante.

Tests attendus :

test_missing_preconditions_are_neutral_not_max_pressure
test_zero_verified_real_preconditions_are_high_pressure
test_missing_preconditions_emit_warning
test_missing_vs_real_zero_pressure_are_distinct
