5.2.1-B — tool requested mais pas de tool evidence

Dans gateway_loop.py, si une définition gateway manque, _run_tool_phase() ajoute une warning et continue. Puis, si aucun enriched_refs n’est créé, la règle de citation pass-2 ne s’applique pas. Résultat possible : le forward pass demande un outil, l’outil n’est pas exécuté, et une claim pass-2 pourrait être acceptée sans preuve outillée nouvelle.

Patch recommandé :

If forward.requested_tools is non-empty
AND no new tool evidence refs are produced
THEN pass-2 accepted claims related to that event must be demoted
OR the report status must become blocked/diagnostic_only with a blocker.

Concrètement :

missing gateway definition:
  blockers += "requested tool had no approved gateway definition"

tool requested but all tool calls blocked/quarantined:
  blockers += "requested tool evidence unavailable"

tool requested but no new [E.tool_output.*] ref:
  pass-2 claims cannot become verification_candidate

Phase 5.3 — objectif

Phase 5.3 doit mesurer si le gateway-grounded verifier loop améliore vraiment la vérification.

La question :

Le passage par le gateway local améliore-t-il :
- la fidélité des citations ?
- l’acceptation/rejet correct des claims ?
- le rejet des contradictions outils ?
- la détection des unsupported claims ?
- la robustesse aux sorties outil hostiles ?

La question n’est pas :

Est-ce que le code est merge-safe ?
Est-ce que OIDA-code prédit la prod ?
Est-ce que total_v_net peut être officiel ?
Est-ce que MCP peut être activé ?

NIST insiste sur les processus TEVV — test, evaluation, verification and validation — tout au long du cycle de vie AI, avec des métriques et méthodes documentées après la fonction MEASURE. C’est exactement le rôle de Phase 5.3 : mesurer proprement le comportement du verifier, pas faire une annonce produit.

5.3-F — private holdout contamination policy

Chaque cas doit porter :

provenance:
  synthetic
  private_trace
  private_repo
  public_low
  public_high

contamination_risk:
  synthetic
  private
  public_low
  public_high

Règle :

public_high excluded from headline metrics
private cases never committed if they contain private code

Ce point est important parce que les benchmarks publics finissent vite par devenir des cibles d’optimisation ou de contamination ; les évaluations de coding agents avec tests peuvent aussi compter comme “correctes” des solutions plausibles mais comportementalement divergentes, ce qui justifie de garder un holdout privé et une analyse d’échec séparée.
