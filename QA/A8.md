Concrètement, le cas testé par Claude est :

src/app.py changed
src/app.py imports src/db.py

Dans ce sens, ça marche, parce que build_dependency_graph() scanne le fichier changé app.py, voit qu’il importe db.py, et peut créer l’edge db -> app.

Mais le cas inverse est tout aussi important :

src/db.py changed
src/app.py imports src/db.py

L’impact cone peut bien inclure src/app.py comme imported_by_changed, car build_impact_cone() cherche aussi les importers directs.
Mais le graphe, lui, risque de ne scanner que src/db.py, puisque obligations_to_scenario() reprend request.scope.changed_files et appelle build_dependency_graph(obligations, repo_path, changed_files).

Donc on peut avoir :

obligations:
  db ✓
  app ✓

impact_surface:
  db ✓
  app ✓

graph:
  db -> app ✗ manquant
