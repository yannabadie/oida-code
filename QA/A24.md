Phase 4.7 doit répondre à une question étroite :

Un vrai provider externe peut-il respecter les contrats OIDA-code sur calibration_v1, sans fuite de secrets, sans violation de schéma, sans citations manquantes, sans champs officiels, et sans dégradation silencieuse par rapport au replay baseline ?

- ce provider est meilleur qu’un autre
- OIDA-code prédit les incidents de production
- le code est merge-safe
- V_net est officiel
- debt_final est officiel
- corrupt_success est officiel

On reste sur une régression de conformité contractuelle, pas sur un benchmark public.

4.7.1 — provider baseline workflow

Créer :

.github/workflows/provider-baseline.yml
Déclenchement

Uniquement :

on:
  workflow_dispatch:

Interdit :

push
pull_request
pull_request_target
schedule

Pourquoi : un provider externe implique des secrets et des coûts. Il ne doit jamais tourner automatiquement, encore moins sur PR fork. GitHub Security Lab rappelle que pull_request_target donne accès aux secrets et aux permissions du dépôt cible, ce qui en fait une surface dangereuse avec du code non fiable.

Permissions
permissions:
  contents: read

Pas de :

security-events: write
actions: write
checks: write
contents: write
Inputs
inputs:
  provider-profile:
    description: "deepseek | kimi | minimax"
    required: true
    default: "deepseek"

  max-provider-cases:
    description: "Maximum external cases to run"
    required: true
    default: "4"

  compare-replay:
    description: "Run replay baseline before provider"
    required: true
    default: "true"
Secrets

Le workflow ne doit jamais contenir une clé en dur.

Il doit mapper les secrets uniquement via secrets.*, par exemple :

env:
  DEEPSEEK_API_KEY: ${{ secrets.DEEPSEEK_API_KEY }}

Et le script doit appeler ::add-mask:: si une valeur sensible est manipulée dynamiquement. GitHub documente add-mask comme mécanisme pour empêcher une valeur sensible d’apparaître dans les logs.

4.7.2 — provider order
Provider primaire

Commencer par :

DeepSeek

Raison : la documentation officielle DeepSeek indique une base URL OpenAI-format https://api.deepseek.com, un contexte 1M, une sortie JSON, des tool calls, et des prix affichés par million de tokens. C’est le meilleur premier candidat pour tester le chemin OpenAI-compatible sans multiplier les variables.

Providers secondaires

Ensuite seulement :

MiniMax
Kimi / Moonshot

MiniMax documente explicitement une API compatible OpenAI, avec OPENAI_BASE_URL=https://api.minimax.io/v1 et appel via l’OpenAI SDK ; ses modèles M2.x listés ont aussi de grands contextes.

Moonshot/Kimi est supporté côté écosystème et LiteLLM documente le provider Moonshot avec base URL globale https://api.moonshot.ai/v1 et variable MOONSHOT_API_KEY, mais je le mettrais après DeepSeek si l’objectif est un premier run simple et peu ambigu.

Règle

Phase 4.7 peut être validée avec un seul provider réel si le budget API est limité.

Ordre recommandé :

1. DeepSeek, max 4 cases
2. DeepSeek, max 8 cases si tout est propre
3. MiniMax, max 4 cases
4. Kimi, max 4 cases
4.7.3 — calibration run

Le workflow doit exécuter :

python -m oida_code.cli calibration-eval datasets/calibration_v1 \
  --llm-provider replay \
  --out .oida/provider-baseline/replay

Puis, si le provider est explicitement activé :

python -m oida_code.cli calibration-eval datasets/calibration_v1 \
  --llm-provider openai-compatible \
  --provider-profile deepseek \
  --api-key-env DEEPSEEK_API_KEY \
  --max-provider-cases 4 \
  --out .oida/provider-baseline/deepseek

4.7.4 — métriques à comparer

Comparer replay vs provider réel sur :

schema_valid_rate
invalid_json_rate
schema_violation_rate
missing_citation_rate
confidence_cap_violation_rate
forbidden_claim_rate
official_field_leak_count
evidence_ref_precision
evidence_ref_recall
safety_block_rate
fenced_injection_pass_rate
provider_unavailable_count
timeout_count

Règles bloquantes :

official_field_leak_count > 0 => fail
unredacted secret in logs/artifacts => fail
provider default-on => fail
provider on PR/fork => fail
invalid JSON not handled cleanly => fail
schema violation not rejected => fail
missing citations accepted silently => fail

Règles non bloquantes :

invalid_json_rate > 0
missing_citation_rate > 0
schema_valid_rate < replay

Ces points ne bloquent pas s’ils sont correctement mesurés, car Phase 4.7 sert justement à apprendre comment les providers réels échouent.
