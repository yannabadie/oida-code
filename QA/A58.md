## Decision requested

After ADR-76, cgpro accepted commit `fd34302` and selected the next
high-leverage block: align the front-door CLI/Markdown reviewer path with the
ADR-74 diagnostic-only product compass.

Codex then asked whether to:

- modify the existing `audit --format markdown` renderer;
- add a new `review` / `diagnose` alias;
- do both;
- or stop for docs-only work.

## cgpro decision

cgpro selected `modify_existing_renderer`.

Rationale:

- `audit --format markdown` is already a supported front-door path, so leaving
  it with legacy verdict language while adding a second alias would keep the
  unsafe surface alive.
- JSON/SARIF compatibility should remain unchanged; only the human Markdown
  renderer and CLI help text change.
- The `repair` command remains visible for compatibility, but its help must
  make clear it is not a front-door path and does not modify code.

## Accepted scope

- Change `src/oida_code/report/markdown_report.py` to render diagnostic-only
  Markdown.
- Update `src/oida_code/cli.py` top-level help, `audit` help, `--format`,
  `--fail-on`, and `repair` help/error wording.
- Add a 10-minute diagnostic CLI quickstart.
- Add Phase 6.e report artifacts and regression tests.
- Update README, product strategy, project status, backlog, decision log,
  progress, and Codex context.

## Forbidden scope

- No new CLI alias.
- No JSON schema, SARIF schema, `AuditReport`, `ReportSummary`, or internal
  verdict enum migration.
- No `resolve_verdict`, `fail-on`, deterministic tool execution, or command
  behavior change.
- No calibration seed index, candidate partition, clone-helper, action default,
  provider runtime, MCP runtime, or gateway default change.
- No direct provider call.
- No merge-readiness, production-readiness, bug-free, security-verification,
  public-benchmark, or autonomous-repair claim.
