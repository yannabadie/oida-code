Phase 4.5 — CI / GitHub Action integration

Phase 4.5 doit être divisée en deux surfaces :

4.5-A — internal CI for oida-code itself
4.5-B — reusable GitHub Action / workflow for target repos

Et non :

production GitHub App
public check-run product
official merge-safe verdict

GitHub Actions recommande de limiter les permissions du GITHUB_TOKEN au strict nécessaire ; donc 4.5 doit partir avec permissions: contents: read par défaut, puis n’ajouter security-events: write que dans le job qui upload éventuellement le SARIF.

4.5-A — CI interne du repo

Créer :

.github/workflows/ci.yml

Objectif : sécuriser les gates déjà utilisés localement.

Jobs :

lint:
  ruff check src/ tests/ scripts/

typecheck:
  mypy src/ scripts/

test:
  pytest -q

calibration:
  python scripts/run_calibration_eval.py
  ou oida-code calibration-eval datasets/calibration_v1

security-smoke:
  vérifie qu’aucun provider externe n’est appelé par défaut

Règles :

permissions:
  contents: read

Pas de secrets dans ce workflow.

Pas de provider externe.

Pas de Qwen.

Pas de MCP.

Pas de PyPI.

4.5-B — Action réutilisable

Créer soit :

action.yml

pour une composite action, soit :

.github/workflows/oida-code.yml

comme template de workflow.

Je recommande d’abord composite action + exemple workflow, parce que c’est plus simple qu’une GitHub App et plus facile à installer.

Inputs minimaux :

inputs:
  repo-path:
    default: "."
  base-ref:
    default: "origin/main"
  intent-file:
    required: false
  output-dir:
    default: ".oida"
  upload-sarif:
    default: "false"
  fail-on:
    default: "none"
  surface:
    default: "impact"
  enable-shadow:
    default: "false"
  llm-provider:
    default: "replay"

Sorties :

.oida/request.json
.oida/report.json
.oida/report.md
.oida/report.sarif
.oida/calibration/metrics.json

Le workflow doit écrire un résumé lisible dans :

$GITHUB_STEP_SUMMARY

Et uploader les artefacts via :

actions/upload-artifact
SARIF : optionnel mais utile

OIDA produit déjà du SARIF. GitHub permet d’uploader des fichiers SARIF tiers via github/codeql-action/upload-sarif, et GitHub Code Scanning attend du SARIF 2.1.0.

Donc :

permissions:
  contents: read
  security-events: write

uniquement pour le job SARIF.

Ne mets pas security-events: write au workflow entier si seul le job SARIF en a besoin.

Ne pas faire de Checks API en 4.5

Pas de GitHub App maintenant.

GitHub précise que l’écriture des check runs via l’API REST est réservée aux GitHub Apps ; les OAuth apps et utilisateurs authentifiés peuvent lire les check runs, mais pas créer de check suite/run.

Donc Phase 4.5 doit utiliser :

- artifacts
- step summary
- SARIF upload optionnel

Pas :

- check runs custom
- annotations Checks API custom

Les check runs viendront plus tard avec une vraie GitHub App.

Secrets et provider externe

Par défaut :

llm-provider = replay

Si l’utilisateur active un provider externe :

env:
  DEEPSEEK_API_KEY: ${{ secrets.DEEPSEEK_API_KEY }}

Il faut masquer toute valeur dynamique avec ::add-mask::; GitHub documente explicitement add-mask pour éviter que des données sensibles apparaissent dans les logs.

Règles :

- aucun provider externe sur pull_request venant d’un fork
- provider externe seulement sur workflow_dispatch ou branche protégée
- pas de secret dans artifacts
- pas de prompt complet contenant secret dans logs
- pas de raw provider response en artifact sauf debug-redacted explicite

ADR-30 — CI and GitHub Action integration before GitHub App

Contenu :

Decision:
  Phase 4.5 adds CI gates and a reusable GitHub Action surface.
  It does not add a GitHub App, custom check runs, or official production verdicts.

  - default permissions read-only
  - SARIF upload optional and isolated
  - provider external opt-in only
  - secrets via GitHub secrets only
  - no provider on fork PRs
  - artifacts + step summary as first UX

Rejected:
  - default external provider
  - custom Checks API without GitHub App
  - exposing official V_net/debt/corrupt_success
  - publishing PyPI stable
  - production merge-safe labels
