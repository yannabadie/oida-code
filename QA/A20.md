Choix fournisseur recommandé

Je recommande de commencer par un OpenAI-compatible provider adapter, pas par un provider custom par modèle.

Pourquoi : DeepSeek, Kimi/Moonshot et MiniMax documentent chacun une API compatible OpenAI Chat Completions, donc un adaptateur unique peut couvrir plusieurs fournisseurs en changeant base_url, model et api_key_env. DeepSeek annonce une compatibilité OpenAI/Anthropic dans ses docs, Kimi indique une compatibilité OpenAI Chat Completions avec base_url Moonshot, et MiniMax fournit aussi une API OpenAI-compatible.

Je ne prendrais pas xAI/Grok comme premier binding si tu veux aller vite avec Chat Completions, parce que leur documentation présente Chat Completions comme endpoint legacy et recommande plutôt l’API Responses pour les nouvelles capacités. Ça ne veut pas dire “pas xAI”, seulement “pas le premier si l’objectif est un adapter simple”.

Sécurité Phase 4.4

Avant tout vrai appel externe :

1. Repo + history secret scan.
2. GitHub push protection enabled if possible.
3. No key in config.
4. No key in fixtures.
5. No key in reports.
6. No key in exception strings.
7. No raw prompt with secrets.
8. No tool output containing secret-like paths sent to model.

GitHub décrit la push protection comme une défense qui bloque les pushes contenant des secrets avant qu’ils n’atteignent le dépôt ; c’est exactement le bon garde-fou à ajouter maintenant que le projet entre dans une phase avec clés réelles.

Côté agent, on garde les règles OWASP déjà suivies : least privilege, permission par outil, séparation des trust levels, et autorisation explicite pour les opérations sensibles. OWASP liste aussi les attaques spécifiques aux agents, notamment thought/observation injection, tool manipulation et context poisoning ; donc tout contenu reçu d’un provider ou d’un outil reste une donnée non fiable.
