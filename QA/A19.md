  request.max_runtime_s,
  remaining_global_budget_s

if remaining_global_budget_s <= 0:
  status = "blocked"
  executor not invoked

Phase 4.3 — objectif

Phase 4.3 ne doit pas prouver que OIDA-code prédit le succès réel.

Phase 4.3 doit définir comment mesurer proprement la qualité des estimateurs, du verifier forward/backward et de la boucle outillée, sans retomber dans le piège de Phase 3 : un signal apparemment bon mais confondu par la longueur des sessions et un proxy de succès faible. Le rapport Phase 3 avait justement montré que le progress_rate était mécaniquement lié à la longueur de session et que l’outcome commits > 0 était tautologique.

Le bon objectif :

Design a calibration dataset and scoring protocol for:
- estimator quality
- evidence citation fidelity
- forward/backward claim validation
- tool-grounded contradiction handling
- shadow pressure ordering

Without:
- official V_net
- official debt_final
- official corrupt_success
- predictive claims about production success

Le design doit s’inspirer de trois leçons externes :

Les benchmarks de code doivent utiliser des signaux exécutables de type fail-to-pass et pass-to-pass quand ils prétendent mesurer une correction de bug ; SWE-bench décrit ce principe, et SWE-bench Multilingual rappelle que les F2P vérifient la correction tandis que les P2P vérifient l’absence de régression.
SWE-bench Verified est devenu problématique comme signal frontier : OpenAI dit avoir cessé de le rapporter, en citant contamination et tests défectueux ; leur audit mentionne qu’au moins 59,4 % des problèmes audités dans un sous-ensemble avaient des défauts importants. Donc on ne doit pas bâtir OIDA-code sur un benchmark public saturé sans garde anti-contamination.
ProdCodeBench est plus proche de ce qu’il faut : il décrit une méthodologie production-derived avec prompt réel, code change commité, tests fail-to-pass, validation de pertinence des tests et multi-run stability checks.

Phase 4.3-C — Taille du pilote

Ne construis pas tout de suite un benchmark massif.

Je veux un pilot calibration set de 32 cas :

8 claim_contract
8 tool_grounded
6 shadow_pressure
6 code_outcome avec F2P/P2P
4 safety_adversarial

Pourquoi 32 : assez pour détecter les erreurs de protocole, pas assez pour prétendre faire de la statistique produit. La suite pourra monter à 100+ après stabilisation.

Répartition attendue :

1. claim_contract

Cas qui testent l’agrégateur sans outils réels :

- forward + backward support -> accepted
- forward only -> unsupported
- backward missing negative path -> unsupported
- unknown evidence ref -> rejected
- forbidden phrase -> rejected
- confidence cap exceeded -> rejected
- benefit missing intent -> unsupported
- repair_needed supported but diagnostic only
2. tool_grounded

Cas avec run-tools :

- ruff finding contradicts claim
- mypy finding contradicts claim
- pytest scoped pass supports precondition
- pytest timeout -> uncertainty
- tool_missing -> uncertainty
- semgrep requested but no adapter -> blocked
- path traversal blocked
- secret-like path blocked
3. shadow_pressure

Cas de pression relative :

- clean_success -> low
- migration_without_rollback -> medium/high
- corrupt_plausible_success -> high but not official
- supportive-only graph -> integrity pressure, not debt pressure
- constitutive graph -> debt pressure can propagate
- missing grounding model -> neutral warning, not max pressure
4. code_outcome

Cas avec repo mini et tests :

- bug fix with F2P pass, P2P pass
- regression: F2P pass, P2P fail
- no fix: F2P fail
- negative path missing
- rollback/idempotency missing
- auth guard missing

Ici, l’évaluation doit être testée avec exécution répétée :

stability_runs = 3
flaky if any run disagrees
flaky cases excluded from calibration metrics
5. safety_adversarial

Cas sécurité :

- prompt injection in code comment
- prompt injection in tool output
- forged evidence id
- attempt to close OIDA_EVIDENCE fence

Les recommandations OWASP justifient ce bloc : les agents sont exposés à prompt injection directe/indirecte, abus d’outils, privilege escalation et exfiltration ; OWASP recommande aussi de séparer clairement instructions et données dans les prompts.

Phase 4.3-F — Anti-contamination et provenance

Chaque cas doit avoir :

class CalibrationProvenance(BaseModel):
    source: Literal["synthetic", "hand_seeded", "private_trace", "public_repo"]
    created_by: Literal["human", "script", "mixed"]
    source_commit: str | None = None
    public_url: str | None = None
    contamination_notes: str = ""

Règle :

public benchmark task reused -> contamination_risk = public_high
public_high cases excluded from headline metrics

La raison est simple : les benchmarks publics peuvent devenir des benchmarks de mémorisation plutôt que de compétence. C’est exactement l’objection soulevée par OpenAI contre SWE-bench Verified en 2026.

Phase 4.3-G — Pas de seuils produits

Interdit en Phase 4.3 :

- threshold tuning for production
- “pressure > X means corrupt_success”
- “accuracy > Y means ready”
- “verified means merge safe”

Autorisé :

- relative ordering checks
- claim-level accept/reject/unsupported correctness
- tool contradiction rejection
- safety block correctness
- F2P/P2P execution sanity on seeded cases

La calibration doit rester au niveau :

Does the measurement pipeline behave as intended on controlled cases?

Pas :

Does this predict production incidents?

C’est cohérent avec NIST AI RMF : le framework structure la gestion du risque autour de fonctions comme govern, map, measure et manage, et la mesure doit être reliée au contexte opérationnel plutôt qu’à une métrique isolée.
