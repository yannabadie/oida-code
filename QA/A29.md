5.1.1 — micro-hardening avant intégration

J’ai relu LocalDeterministicToolGateway.run(). Il fingerprint la gateway_definition, cherche l’approbation via tool_id, puis exécute l’adapter via request.tool. Ça veut dire qu’il faut verrouiller explicitement que :

request.tool == gateway_definition.tool_name

Sinon, un appel mal câblé pourrait auditer/fingerprinter une définition ruff, mais exécuter pytest parce que request.tool dit autre chose. Le code actuel appelle bien get_adapter(request.tool) après les étapes admission/fingerprint, donc ce contrôle doit être ajouté avant l’exécution.

Patch 5.1.1 demandé :

1. Block if request.tool != gateway_definition.tool_name.
2. Emit audit event with policy_decision="block".
3. Return VerifierToolResult(status="blocked").
4. Put the reason in blockers, not only warnings.

Deuxième point : _blocked_result() retourne aujourd’hui le motif dans warnings, mais pas dans blockers. Pour les étapes suivantes, un tool bloqué doit être une vraie condition bloquante côté boucle verifier. Mets donc :

VerifierToolResult(
    tool=request.tool,
    status="blocked",
    warnings=(reason,),
    blockers=(reason,),
    ...
)

Phase 5.2 — objectif

Phase 5.2 doit brancher le gateway dans la boucle de vérification existante.

Objectif :

forward/backward verifier
→ proposes tool requests
→ gateway validates/admits/fingerprints/runs tools
→ tool results become EvidenceItems
→ second verifier pass cites tool evidence
→ aggregator accepts/rejects claims

Non-objectifs :

- pas de MCP
- pas de JSON-RPC
- pas de tools/list
- pas de tools/call
- pas de provider tool-calling
- pas de remote tools
- pas de write tools
- pas de network egress
- pas de GitHub App
- pas de Checks API
- pas de official total_v_net
- pas de official debt_final
- pas de official corrupt_success

MCP reste volontairement repoussé : le protocole officiel expose des primitives dynamiques comme tools/list et tools/call, ainsi que des notifications quand les tools changent, ce qui est exactement la surface qu’on veut éviter tant que le gateway local n’est pas pleinement intégré.

OWASP liste aussi les risques MCP principaux — tool poisoning, rug pull, tool shadowing, confused deputy, exfiltration via canaux légitimes, permissions excessives — et recommande le pinning cryptographique des définitions, la surveillance des changements, le least privilege et les logs d’invocation.

Créer un runner :

class GatewayGroundedVerifierRun(BaseModel):

Fonction :

def run_gateway_grounded_verifier(
    packet: LLMEvidencePacket,
    *,
    forward_pass1: VerifierProvider,
    backward_pass1: VerifierProvider,
    forward_pass2: VerifierProvider,
    backward_pass2: VerifierProvider,
    gateway: LocalDeterministicToolGateway,

    admission_registry: ToolAdmissionRegistry,
    gateway_definitions: Mapping[ToolName, GatewayToolDefinition],
    audit_log_dir: Path,
    max_tool_calls: int = 5,
) -> GatewayGroundedVerifierRun:
    ...

Flux strict :

Pass 1:
  run forward/backward replay
  collect requested_tools from forward result

Tool phase:
  convert specs to VerifierToolRequest
  run each through LocalDeterministicToolGateway
  collect VerifierToolResult.evidence_items

Evidence enrichment:
  append tool EvidenceItems to LLMEvidencePacket
  optionally add deterministic SignalEstimate for tool failures

Pass 2:
  run forward/backward again with enriched packet

Aggregate:
  existing aggregate_verification decides

Pas de boucle infinie.

Budget :

max_passes = 2
max_tool_calls = 5
no retries
no external verifier provider by default

5.2-D — deterministic tool contradictions restent prioritaires

L’agrégateur actuel rejette une claim si une estimation tool-grounded sur le même event a value < 0.5, avec source tool, static_analysis ou test_result.

Phase 5.2 doit transformer les résultats gateway en signaux déterministes exploitables par cette règle.

Mapping minimal :

VerifierToolResult.status == "failed":
  SignalEstimate(
    field="operator_accept" or "tests_pass",
    value=0.0,
    confidence=0.8,
    source="tool",
    evidence_refs=(...)
  )

status in {"error", "timeout", "tool_missing", "blocked"}:
  warning/blocker
  not negative proof

status == "ok":
  weak positive

Cette distinction reste alignée avec la logique Phase 4.2 : failed est un vrai signal négatif, alors que tool_missing, timeout ou error sont de l’incertitude, pas une preuve que le code est mauvais. Le contrat VerifierToolResult le documente déjà.
