5.0-B — Trust boundaries

Définir une matrice de confiance :

trusted:
  - OIDA deterministic adapters
  - local schema validators
  - frozen Pydantic contracts
  - existing tool registry allowlist

semi-trusted:
  - provider structured output after validation
  - replay fixtures
  - calibration labels

untrusted:
  - repo code under audit
  - tool output
  - MCP tool descriptions
  - MCP input schemas
  - MCP resource contents
  - provider prose
  - GitHub PR metadata
  - branch names
  - workflow inputs

Règle centrale :

Un tool MCP ne devient jamais trusted parce qu’il est bien décrit.
Il devient seulement "eligible for execution" si son identité, son schéma,
son hash, sa policy et son budget passent les contrôles.

Cette distinction est alignée avec la trajectoire déjà prise : Phase 4.0/4.1 a traité les evidence packets et les claims comme données contrôlées, Phase 4.2 a imposé une registry locale read-only, et Phase 4.9 a rendu les artefacts lisibles sans créer de faux verdict. Le projet a déjà corrigé les pièges de validation confondue détectés en Phase 3, où certains signaux étaient mécaniquement liés à la longueur des sessions plutôt qu’à la qualité réelle.

5.0-D — Tool schema pinning

Créer une spec de pinning, pas une implémentation.

Format proposé :

{
  "tool_id": "github.read_pr",
  "server_id": "github_mcp",

  "transport": "stdio|http",
  "schema_version": "2026-04-30",

Règles :

- toute modification d’output schema => review
- toute hausse de permission => rejet automatique
- tout tool sans hash => interdit

Pourquoi : MCP permet des listings dynamiques et des notifications quand les tools changent ; sans pinning, un serveur initialement approuvé peut changer son comportement ou sa description après coup.

5.0-E — Provider tool-calling policy

Même si MCP reste bloqué, il faut aussi cadrer le provider tool-calling.

Pydantic-AI montre bien pourquoi ce sujet doit rester sous contrôle : ses function tools permettent au modèle d’exécuter des actions ou de récupérer de l’information, et ses toolsets peuvent être composés, remplacés dynamiquement, ou fournis via MCP / third-party toolsets. C’est puissant, mais c’est exactement la surface que le projet a volontairement évitée jusqu’ici.

Règles design :

Provider tool-calling is forbidden by default.

Future enablement requires:
  - same ToolPolicy as local verifier tools
  - no raw provider-defined tools
  - no unvalidated provider tool args
  - no provider-issued command strings
  - deterministic adapter owns execution
  - aggregator still has final authority

Un provider ne doit jamais avoir le droit de dire :

run this shell command
call this MCP server
write this file
upload this artifact
mark this PR safe

Il peut seulement produire :

requested_tool_spec

qui passe ensuite par :

policy validator
→ deterministic adapter
→ evidence item
→ forward/backward verifier
→ aggregator
