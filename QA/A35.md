Phase 5.8-B — exécution workflow

Chaque cas doit être lancé via workflow_dispatch ou push contrôlé. GitHub confirme que workflow_dispatch permet de lancer manuellement un workflow depuis l’API, la CLI ou l’UI GitHub, ce qui est exactement la bonne surface pour cette phase opérateur.

Invocation attendue :

enable-tool-gateway: "true"
gateway-bundle-dir: "<bundle explicite>"
gateway-output-dir: ".oida/operator-soak/<case_id>"
gateway-fail-on-contract: "false"
llm-provider: "replay"

Garde-fous :

permissions: contents: read
no pull_request_target
no fork PR
no external provider
no secrets
no MCP
no write/network

GitHub recommande de donner au GITHUB_TOKEN les permissions minimales nécessaires et de garder les permissions par défaut en lecture seule quand possible.

Phase 5.8-C — labels humains obligatoires

Pour chaque cas terminé :

label.json
ux_score.json

Le label doit être écrit par un humain, pas par LLM.

Labels autorisés :

useful_true_positive
useful_true_negative
false_positive
false_negative
unclear
insufficient_fixture

Chaque label.json doit contenir une justification de 3–10 lignes.

Le script run_operator_soak_eval.py lit les sidecars fiche, label, ux_score, puis écrit reports/operator_soak/aggregate.json et .md; il ne lance pas de provider externe, ne lit pas de secret et ne produit pas de verdict produit.

Phase 5.8-D — UX scoring

Chaque cas doit répondre aux 4 questions :

summary_readability: 0|1|2
evidence_traceability: 0|1|2
actionability: 0|1|2
no_false_verdict: 0|1|2

À vérifier manuellement :

- le Step Summary suffit-il à comprendre le résultat ?
- summary.md explique-t-il ce qui est prouvé / non prouvé ?
- grounded_report.json est-il auditable ?
- le rapport évite-t-il tout faux verdict produit ?

GitHub fournit $GITHUB_STEP_SUMMARY pour afficher du Markdown directement dans la page de résumé du run, ce qui en fait la bonne surface UX pour ce type de diagnostic opérateur.

Phase 5.8-F — pas de scheduled/background agent

Ne pas créer de background agent maintenant.

Si un rappel est vraiment utile, créer plutôt :

issue GitHub: Phase 5.8 operator soak tracking

ou une note dans progress.md.

Un workflow schedule peut être utilisé plus tard uniquement comme rappel read-only, mais GitHub documente que les workflows planifiés peuvent être retardés ou même abandonnés en période de forte charge, et qu’ils ne tournent que sur la branche par défaut.

Donc :
