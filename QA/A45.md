## What this means for Phase 6.1'e

cgpro's verdict translates into a 4-step Phase 6.1'e ordering:

### Step 1 — runtime-loader acceptance guard (front-loaded)

The Phase 6.1'b acceptance criterion (`validate_gateway_bundle ok`)
caught file-presence issues but missed runtime Pydantic-loader
shape mismatches. The fix that ADR-57 already shipped (3 generator
fixes + 2 updated tests) handled the symptoms; 6.1'e step 1 adds
the GUARD: a structural test that loads every generated bundle's
8 files through their target Pydantic contracts (LLMEvidencePacket,
ToolPolicy, ToolAdmissionRegistry, GatewayToolDefinition,
ForwardVerificationResult, BackwardVerificationResult lists). This
makes the Phase 6.1'b acceptance "file-presence valid AND
runtime-loader valid" so a future skeleton change cannot reintroduce
the same class of bug.

### Step 2 — pin 2 more train cases (N_pinned 3 → 5)

Currently 1 train + 2 holdout (ratio 0.67). Target: 3 train + 2
holdout (ratio 0.40, inside [0.20, 0.40] enforcing band). The
2 new train cases must come from the EXISTING 46 inclusions
(no new collection), be authored manually with full Tier-3
pedigree, and be partition-pinned with `partition: train` BEFORE
the seed_008 real-checkout work begins (so the generator can
inform its target-checkout helper from train cases without
violating ADR-56 holdout discipline).

Candidate train cases to author (from the 46 inclusions, picking
non-backport non-release-prep cases per ADR-54 §"Selection-effect
caveat"):

* `seed_062_simonw_sqlite_utils_690` — "Allow callable references
  in sqlite-utils convert" (feature addition with test)
* `seed_109_encode_httpx_3690` — "Add `.wait_ready` to parser"
  (feature addition with test)
* `seed_103_samuelcolvin_watchfiles_301` — "return file deleted
  instead of raising an exception" (behavior change with test)
* (any of) the smaller pydantic bugfixes — `seed_030`, `seed_031`,
  `seed_034`

The selection criterion: small diff, has a test, clear
single-claim semantics. Pick 2 of these.

### Step 3 — seed_008 real target checkout

Clone `pytest-dev/pytest@480809ae02a97344e68e52eb015e68b840f2e05c`,
point `--repo-root <clone>`, regenerate the bundle, regenerate
LLM replays, run `verify-grounded`. Expected outcome:
`status=verification_candidate` with `accepted_claims:
["C.cli_version_flag.repair_needed"]`. This is the first
claim-supporting outcome in the calibration_seed lane.

Authoring constraints:

* The clone helper is a NEW manual-lane script
  (`scripts/clone_target_at_sha.py`, `MANUAL_EGRESS_SCRIPT = True`,
  refuses without `--manual-egress-ok`). The script ONLY
  shallow-clones a public repo at a specific SHA into a temp
  workspace; never modifies the target, never pushes, never
  comments.
* The clone is keyed on `(repo_url, head_sha)` and idempotent —
  re-running on the same SHA reuses the existing clone.
* The clone is NOT committed; the round-trip evidence (the
  resulting `grounded_report.json`) is committed under
  `reports/phase6_1_e/round_trip_outputs/seed_008/`.

### Step 4 — holdout checkouts (under freeze rule)

ONLY AFTER step 3 succeeds AND step 2 has reached N_pinned=5,
run the same flow for `seed_065` and `seed_157`:

* `seed_065`: clone `simonw/sqlite-utils@<head_sha>`, run pipeline,
  expect `status=verification_candidate`.
* `seed_157`: clone `hynek/structlog@<head_sha>`, run pipeline,
  expect `status=verification_candidate`.

**Freeze rule enforced:** if either holdout produces
`diagnostic_only` or `blocked` instead of `verification_candidate`,
the operator must NOT modify the generator, the LLM-author
script, the bundle generation logic, or the verifier in response.
Two acceptable outcomes:
1. Demote the case to `train` (with `partition_pinned_at` updated
   and a new holdout pinned to replace it).
2. Document the failure as a real generalisation gap and surface
   it in the Phase 6.1'e report — the holdout's purpose is to
   reveal exactly this.

### Step 5 (optional) — multi-provider replay panel

Per QA/A44 §"Multi-provider panel sizing": run the LLM-author
script across 3-5 providers (DeepSeek + Grok + MiniMax minimum)
and report cross-model agreement on the same bundles. Not
blocking for Phase 6.1'e completion.

## Acceptance for Phase 6.1'e

Per cgpro's overall recommendation:

* Phase 6.1'b acceptance criterion is upgraded from
  "validate_gateway_bundle ok" to "validate_gateway_bundle ok AND
  runtime-loader smoke ok" (step 1).
* Holdout-ratio guard becomes ENFORCING (not informational) once
  N_pinned ≥ 5 (achieved via step 2).
* `verification_candidate` outcome demonstrated on at least
  seed_008 (step 3); attempted on seed_065 + seed_157 under
  freeze rule (step 4).
