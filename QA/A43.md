# QA/A43 — Phase 6.0 protocol-only closure + Phase 6.1' scope under unavailable external operators

**Verdict: Phase 6.0 closed as protocol-only; Phase 6.1' scoped from AI-tier + Yann-solo dogfood evidence.**

## Verdict détaillé

Phase 6.0 ne doit plus rester indéfiniment en état "accepted with conditions, partial".

Le bon verdict A43 est : **Phase 6.0 closed as protocol-only; external-human beta not_run due unavailable operators; Phase 6.1' may proceed from AI-tier + Yann-solo dogfood, with explicit evidence downgrade.**

Ce n'est pas un rejet de Phase 6.0. C'est la fermeture honnête d'une phase dont le pack, les garde-fous, les agrégats zéro-échantillon, les tests, et l'isolation des lanes sont livrés, mais dont l'objectif humain externe ne peut pas être exécuté avec les ressources disponibles.

QA/A41 demandait 2 à 3 opérateurs externes, 3 à 5 repos/PRs contrôlés, et une collecte de feedback humain pour chaque run complété ; QA/A41 autorisait aussi des raisons not_run et une phase partielle, mais pas une attente infinie sans opérateurs.

Le commit `09a79fb` ferme les conditions A42 critiques avant ingestion : wording zéro-échantillon, `feedback_channel="human_beta"` requis, identifiants non vides, booléens explicites, isolation du chemin adversarial, et tests structurels additionnels.

Le commit `7299011` ajoute une lane `ai_adversarial` séparée, avec critiques froides de plusieurs fournisseurs, sans contamination de l'agrégat humain, sans runtime MCP, sans provider tool-calling runtime, et sans changement du défaut gateway.

Donc : **Phase 6.0 est terminée comme protocole, pas réussie comme beta humaine**.

La phrase publique recommandée est :

> Phase 6.0 established the controlled-beta protocol and closed the external-human beta attempt as not_run because no external operators were available. Phase 6.1' uses explicitly separated AI-tier critique plus Yann-solo dogfood as downgraded product-evolution evidence.

Ce statut doit remplacer "human validation still pending" par "external-human validation unavailable / not_run".

A43 doit aussi préciser que le canal humain externe peut être rouvert plus tard, mais qu'il ne bloque plus le travail de réduction de friction.

Le projet ne doit pas prétendre avoir obtenu le signal que QA/A41 cherchait. Le projet peut agir sur un autre signal, moins fort, correctement nommé.

## Ré-cadrage protocole

**Recommandation : option C — RELAX explicit-AI-tier with metric pinning.**

La règle QA/A41 "Do not automate feedback labels with LLM" reste valide pour `feedback_channel: human_beta`. Elle ne doit pas être effacée. Elle doit être interprétée comme : aucun LLM ne peut remplir le formulaire opérateur humain, produire un `operator_label`, ni alimenter `reports/beta/beta_feedback_aggregate.*`.

Mais A43 doit créer un canal séparé : `feedback_channel: ai_tier` ou `critique_channel: ai_adversarial`.

Ce canal peut produire :
- un `agent_label`
- des findings C1..Cn
- des convergences, des divergences, et des recommandations candidates

Ce canal **ne peut pas** produire :
- `operator_label`
- incrémenter `operators_total`
- modifier les métriques humaines
- être résumé comme "beta feedback" sans le qualificatif AI-tier

Le commit `7299011` suit déjà cette logique : l'agrégat dit que les critiques ne remplacent pas le feedback humain, que les agents ne choisissent pas Phase 6.1, et que les critiques sont des premières opinions de lecteurs froids. A43 doit formaliser ce qui était encore implicite : quand le canal humain est impossible, le canal AI-tier devient admissible pour prioriser des patches, pas pour valider l'outil.

### Ce que `docs/project_status.md` doit exposer

Quatre compteurs publics séparés :

```
human_tier_external_runs: 0
ai_tier_cold_reader_runs: N
yann_solo_dogfood_runs: N
external_human_beta_status: not_run_unavailable_operators
```

### Schéma minimal AI-tier

`reports/ai_adversarial/schema.md` doit lister :

- `agent_run_id`
- `agent_label`
- `provider`
- `model_id`
- `pin_date`
- `input_scope` (commit hash + paths reviewed)
- `finding_ids`
- `convergence_level`
- `rejected_suggestions`
- `human_tier_contamination: false` (assertion explicite)

### Métriques AI-tier — pas opérateur

Les métriques AI-tier doivent rester **qualitatives ou ordinales**. Ne pas convertir trois agents en pseudo-échantillon utilisateur. Ne pas publier `operator_usefulness_rate` pour AI-tier. Ne pas reprendre les axes 0/1/2 du formulaire humain tels quels.

Créer plutôt :

```
ai_findings_total
ai_convergence_3of3_count
ai_convergence_2of3_count
ai_rejected_contract_violation_count
ai_actioned_findings_count
ai_regression_recheck_passed_count
```

### Honesty statement A43

> AI-tier feedback is not human operator feedback. It is allowed only as separated cold-reader critique and regression pressure when external-human recruitment is unavailable.

Ce re-cadrage est plus honnête que l'option A (qui condamnerait le projet à l'immobilisme malgré un signal utile). Il est plus sûr que l'option B (qui restait implicite), parce que l'option C force la publication du statut exact : AI-tier + Yann solo, human-tier vide.

## Phase 6.1 scoping

**Recommandation : option Z — Phase 6.1' = convert ai_adversarial signal into product evolution.**

Ne pas choisir X tel quel. Ne pas différer Y indéfiniment.

La convergence C1 est forte : les trois lecteurs froids réussis ont identifié le bundle authoring comme friction dominante, avec absence de schéma, absence de walkthrough par fichier, absence d'exemple non-self-audit, et absence de générateur. Mais l'agrégat lui-même dit que C1 ne déclenchait pas encore Phase 6.1 sous la règle initiale "mesurer d'abord avec humains".

A43 change la règle parce que la mesure humaine externe est indisponible. Le changement ne doit pas être : "les agents ont remplacé les humains". Le changement doit être : "le projet passe à une phase 6.1' de réduction de friction sous preuve dégradée".

Le scope recommandé pour Phase 6.1' est donc :

> **Build the smallest bundle-authoring aid that can reduce C1 friction while preserving all existing structural pins.**

Ce n'est pas encore un bundle generator complet. C'est une phase de conversion du signal C1 en outil minimal et mesurable.

### Ordre recommandé

1. **Phase 6.0.z** : doc polish C2-C5 (avant tout générateur).
2. **Phase 6.1'a** : bundle schema explainer + un exemple non-self-audit.
3. **Phase 6.1'b** : `prepare-gateway-bundle` skeleton generator.
4. **Phase 6.1'c** : Yann-solo dogfood run on one controlled repo.
5. **Phase 6.1'd** : AI-tier re-run on updated docs and generated bundle.

C2, C3, C4, C5 doivent être traités avant le générateur, parce qu'ils sont petits et réduisent le bruit.

- **C2** : reformuler le signal `verification_candidate` pour éviter une lecture de pseudo-décision.
- **C3** : supprimer ou cadrer l'estimation 30–60 min comme non mesurée.
- **C4** : définir le "verifier loop" avant usage.
- **C5** : documenter le format de claim et les valeurs `allowed_fields`.

### Première implémentation du generator (Phase 6.1'b)

**Entrées acceptées :**
- repo path local
- target commit
- pytest scope
- claim id
- allowed field
- human-written claim summary
- output dir

**Sortie acceptée :**
- arborescence bundle
- fichiers JSON valides
- champs vides explicites à compléter
- checklist humaine
- refus si scope ou champ hors allowlist

**Refus obligatoires :**
- pas de repo distant implicite
- pas de secrets
- pas de PR fork
- pas de write tool
- pas d'egress
- pas de provider call
- pas de génération autonome de label

**Le generator doit aider à écrire un bundle, pas décider que le bundle est bon.**

### Critère de succès Phase 6.1'

Le succès Phase 6.1' ne doit pas être "les agents aiment le nouveau pack". Le succès Phase 6.1' doit être :

- Yann peut produire un bundle plus vite qu'avant
- Le bundle généré passe les validateurs
- L'AI-tier ne retrouve pas C1 sous la même forme
- Aucun garde-fou structurel n'est affaibli
- Le canal humain reste explicitement vide ou séparé

C1 est donc un signal suffisant pour scoper Phase 6.1' autour du bundle generator minimal. **C1 n'est pas un signal suffisant pour déclarer que le generator résout la friction utilisateur.**

## Yann-solo policy

Yann peut exécuter des beta cases solo.

Yann **ne doit pas** remplir `feedback_channel: human_beta` comme s'il était opérateur externe. La raison est simple : QA/A41 cherchait précisément un usage hors Yann / Claude / cgpro, et listait l'absence de diversité de workflows humains comme manque à combler.

Yann peut produire un canal séparé :

```yaml
feedback_channel: yann_solo_dogfood
operator_role: project_author
bias_disclosure: author_of_docs_and_protocol
target_repo_familiarity: low|medium|high
bundle_authoring_mode: manual|generated|mixed
```

Ses runs sont humains, mais non externes. Ils valent **plus** que de l'AI-tier pour mesurer la friction réelle de lancement. Ils valent **moins** qu'un opérateur externe pour mesurer compréhension froide, jargon, et faux verdicts cognitifs.

Yann-solo peut remplir une version du formulaire, mais elle doit être stockée hors agrégat humain externe.

**Deux options acceptables :**
- `reports/beta_yann_solo/`
- `reports/beta/internal_dogfood/`

**Interdit :**
- l'ajouter à `operators_total`
- l'appeler external beta
- l'utiliser seul pour clore la validation humaine
- le mélanger aux agents dans un même agrégat

### Ce que Yann-solo mesure bien

- temps réel de préparation bundle
- nombre d'allers-retours schema
- erreurs de champs
- friction `workflow_dispatch`
- clarté des artefacts après exécution
- stabilité des docs quand l'auteur essaie de suivre ses propres instructions

### Ce que Yann-solo mesure mal

- première compréhension conceptuelle
- ambiguïtés que l'auteur corrige mentalement
- confiance excessive dans les termes internes
- risque de pseudo-décision perçue par un lecteur froid

### Règle pratique

- AI-tier convergence 3/3 + Yann-solo friction reproduite = scope Phase 6.1' acceptable
- Yann-solo seul = patch local acceptable, phase boundary insuffisant
- AI-tier seul = patch doc acceptable, implémentation seulement si le problème touche un bloc déjà prédit par QA/A41
- External human later = réouvre human-tier et peut contredire Phase 6.1'

## Nouveaux pièges

13. **Anthropomorphiser les agents.** Un "cold reader" LLM n'est pas un opérateur. Il peut exposer des ambiguïtés textuelles, mais pas mesurer l'installation, la fatigue, la confiance, ou le coût social d'usage en PR.
14. **Transformer la convergence 3/3 en vérité produit.** Trois agents qui lisent le même pack avec un prompt similaire peuvent partager le même biais d'instruction. La convergence est un prior, pas une validation.
15. **Contamination par vocabulaire.** Même avec isolation des chemins, un résumé public peut dire "beta feedback" alors qu'il signifie "AI-tier critique". Le risque de contamination est discursif, pas seulement technique.
16. **Goodhart sur les agents.** Si les docs sont modifiées pour satisfaire les mêmes trois prompts, elles peuvent devenir meilleures pour les LLMs et pas meilleures pour les humains. Chaque re-run doit inclure une question : "ce patch aide-t-il un opérateur ou seulement le critique ?"
17. **Agents trop utiles sur les docs, trop mauvais sur le workflow.** Les LLMs repèrent bien les contradictions et le jargon. Ils ne ressentent pas réellement la friction de `workflow_dispatch`, des artefacts, des chemins locaux, ou des permissions.
18. **Rejet trop large des divergences.** D1 a été correctement rejeté parce qu'il attaquait des pins structurels. Mais toute divergence ne doit pas être rejetée par réflexe. Une divergence peut devenir signal si Yann-solo ou un agent futur la reproduit sous une autre forme.
19. **Faux confort par "aucun token interdit".** L'agrégat distingue déjà token leak et reader-cognition leak. C'est important : une doc peut éviter les mots interdits et quand même pousser un lecteur vers une mauvaise interprétation.
20. **Statut public trop optimiste.** "AI-tier beta completed" serait trompeur. Le bon libellé est "AI-tier cold-reader critique completed".
21. **Yann-solo peut corriger mentalement les trous du bundle.** Il faut l'obliger à noter chaque moment où il utilise une connaissance hors docs. Ces notes sont plus importantes que son score final.
22. **Le generator peut devenir une architecture cachée.** Un helper de bundle peut facilement glisser vers sélection automatique de claim, génération automatique d'evidence, ou décision de scope. A43 doit interdire ces extensions en Phase 6.1'a/b.
23. **Re-run AI sans pinning.** Chaque critique doit pinner provider, model id, date, prompt hash ou prompt path, input docs commit, et output path. Sinon la comparaison avant/après n'a pas de valeur.
24. **Kimi 401 silencieux.** L'échec Kimi doit rester un failure record, pas disparaître du nombre de tentatives. Un provider raté réduit la diversité réelle du panel.
25. **Métriques AI-tier trop proches des métriques humaines.** Si les deux agrégats ont les mêmes noms de champs, quelqu'un les mélangera. Préférer `ai_convergence_*` et `ai_actioned_*`, pas `operator_*`.
26. **Phase 6.1' apostrophe oubliée.** Le prime est important : Phase 6.1' n'est pas la Phase 6.1 prévue par feedback humain. C'est une branche de réduction de friction sous preuve dégradée.
