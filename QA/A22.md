4.5.2-D — traiter les warnings Node 20

Ce n’est pas le bloqueur immédiat, mais le runner affiche aussi les warnings Node 20 sur actions/checkout@v4, actions/setup-python@v5, actions/upload-artifact@v4, et potentiellement upload-sarif. GitHub a annoncé que Node 20 est en dépréciation, que les runners commencent à basculer vers Node 24 par défaut à partir du 2 juin 2026, et recommande de tester avec FORCE_JAVASCRIPT_ACTIONS_TO_NODE24=true ou de mettre à jour les actions vers des versions compatibles.

Je ne bloquerais pas 4.5 là-dessus, mais je créerais un ticket 4.5.3 — Node24 compatibility :

- tester FORCE_JAVASCRIPT_ACTIONS_TO_NODE24=true
- bump actions vers versions Node24-compatible quand disponibles
- vérifier upload-artifact / checkout / setup-python / upload-sarif

La doc GitHub actuelle montre déjà l’usage de upload-sarif pour les outils SARIF tiers et rappelle que l’upload passe par github/codeql-action/upload-sarif; elle montre aussi les permissions security-events: write pour le job d’upload. Ton architecture “SARIF optionnel + permission localisée” reste donc correcte.

Ce que je valide malgré tout

Le design de sécurité GitHub Actions reste bon :

pas de pull_request_target, ce qui évite le scénario classique “pwn request” où un PR forké peut être combiné à un contexte privilégié ; GitHub Security Lab décrit explicitement ce mélange comme dangereux.
les valeurs PR-contrôlées sont passées via env: puis utilisées comme $VAR dans Bash ; c’est exactement la mitigation recommandée par la documentation GitHub pour limiter l’injection dans les scripts inline.
le provider externe reste désactivé par défaut et interdit sur les PR forks.
les champs officiels V_net, debt_final, corrupt_success restent absents.

Donc le problème n’est pas architectural. C’est un problème de premier passage sur runner propre, principalement dependency hygiene.
