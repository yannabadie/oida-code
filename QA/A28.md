Verdict Phase 5.0

Phase 5.0 est validée.

Accepté :

- ADR-35 écrit
- threat model MCP produit
- admission policy produite
- schema pinning / rug-pull design produit
- tool-call execution model produit
- audit-log schema produit
- Pydantic-AI assessment produit

- zéro code MCP
- zéro dépendance MCP
- zéro workflow MCP
- zéro provider tool-calling
- zéro champ officiel OIDA émis
- runs GitHub-hosted verts

Le rappel important : Phase 5.0 est un design de sécurité, pas une implémentation. MCP ajoute une architecture host/client/server, des primitives tools, resources, prompts, des méthodes de discovery comme tools/list, et de l’exécution via tools/call; cette surface dynamique justifie de ne pas ouvrir MCP avant d’avoir des garde-fous runtime sur les outils déjà connus.

Décision Phase 5.1

Je recommande :

Phase 5.1 = local deterministic tool gateway

Et je rejette pour l’instant :

Phase 5.1 = MCP sandbox proof-of-concept

Pourquoi : le projet a gagné en solidité parce qu’il a refusé les faux signaux trop tôt. La Phase 3 avait déjà montré qu’un signal séduisant pouvait être confondu par la longueur des sessions et par un proxy faible ; il faut garder la même discipline pour l’autonomie outillée.

Le verrou V_net / debt_final / corrupt_success reste aussi inchangé : la consultation v0.4.1 posait déjà que ces champs ne doivent pas être exposés tant que les signaux porteurs et la validation ne sont pas fiables.

5.1 — objectif

Phase 5.1 doit transformer les designs Phase 5.0 en garde-fous runtime sans MCP.

Objectif :

Wrap existing deterministic tools behind a gateway that enforces:
- admission policy
- schema fingerprinting
- read-only execution
- no secret access
- no network by default
- audit log per tool call
- evidence output only
- aggregator remains final authority

Non-objectifs :

- pas de MCP runtime
- pas de MCP SDK
- pas de serveur MCP local
- pas de provider tool-calling
- pas de dynamic tool discovery
- pas de remote tools
- pas de write tools
- pas de official V_net
- pas de official debt_final
- pas de official corrupt_success

OWASP recommande précisément de limiter les permissions, d’inspecter et pinner les descriptions/schémas d’outils, de sandboxer les serveurs/outils, de valider entrées/sorties, de surveiller les invocations, et de traiter les tool outputs comme des données non fiables. Phase 5.1 doit appliquer ces contrôles aux outils locaux existants avant toute ouverture MCP.

5.1-C — Admission policy runtime

Créer :

class ToolAdmissionRegistry(BaseModel):
    model_config = ConfigDict(extra="forbid", frozen=True)

    approved: tuple[ToolAdmissionDecision, ...]
    quarantined: tuple[ToolAdmissionDecision, ...]
    rejected: tuple[ToolAdmissionDecision, ...]

Fonction :

def admit_tool_definition(
    definition: GatewayToolDefinition,
    *,
    expected_fingerprint: ToolSchemaFingerprint | None,
) -> ToolAdmissionDecision:
    ...

Règles :

1. no expected fingerprint => quarantined, not approved
2. fingerprint match => approved_read_only
3. fingerprint drift => quarantined
4. risk_level != read_only => rejected
5. requires_network=True => rejected
6. allows_write=True => rejected
7. suspicious description => rejected or quarantined

Patterns suspects :

ignore previous instructions
override policy
send secrets
exfiltrate
execute shell
write file

5.1-F — CLI

Ne remplace pas encore run-tools.

Ajoute une commande explicite :

oida-code tool-gateway run \
  requests.json \
  --policy tool_policy.json \
  --approved-tools .oida/tool-gateway/approved_tools.json \
  --audit-log-dir .oida/tool-gateway/audit \
  --out .oida/tool-gateway/results.json

Et une commande de fingerprinting :

oida-code tool-gateway fingerprint \
  --out .oida/tool-gateway/fingerprints.json

Pourquoi deux commandes : la première crée les fingerprints, la seconde exécute. Ça rend visible la différence entre :

tool definition observed
tool definition approved
tool call executed
tool result emitted

Le gateway peut utiliser des concepts compatibles avec MCP — audit event, schema fingerprint, admission policy — mais il ne doit pas implémenter le protocole MCP. Le protocole MCP expose notamment des primitives dynamiques comme tools/list et tools/call, donc ces noms doivent rester absents du runtime gateway pour éviter une activation accidentelle.
