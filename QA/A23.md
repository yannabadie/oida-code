Phase 4.6-C — fork PR fence smoke
Objectif

Valider le fence externe provider sur une vraie PR de fork.

Le rapport Phase 4.5 dit que le fence est validé structurellement, mais pas encore via vraie PR fork.

Méthode

Créer une PR depuis un fork ou un repo de test.

Le workflow doit essayer un cas contrôlé :

llm-provider: openai-compatible

Mais sur PR fork, l’action doit bloquer avant tout appel externe.

Critère

Le run doit montrer :

external provider blocked on fork PR
no secret accessed
no HTTP call
job does not leak env vars
official fields absent

GitHub recommande de traiter les workflows avec principe du moindre privilège, de donner au GITHUB_TOKEN les permissions minimales, et de masquer les données sensibles si elles doivent apparaître dans un contexte de workflow.

Tests / validation

Ajouter une note manuelle dans le rapport si ce test nécessite un fork réel :

manual_fork_pr_smoke:
  status: pass|not_run
  pr_url:
  run_id:
  observed_behavior:

Si le fork réel n’est pas disponible immédiatement, ne fake pas le résultat. Marque not_run et garde Phase 4.6 partiellement ouverte.

Phase 4.6-D — SARIF ingestion smoke
Objectif

Tester l’upload SARIF contre GitHub Code Scanning, pas seulement la génération locale.

Phase 4.5 dit clairement que le SARIF est généré et shape-testé, mais que l’upload github/codeql-action/upload-sarif n’a pas encore été exercé.

Travail demandé

Créer un workflow manuel :

on:
  workflow_dispatch:
    inputs:
      upload-sarif:
        default: "false"

Quand upload-sarif=true :

permissions:
  contents: read
  security-events: write

Seulement pour le job SARIF.

Le job doit :

1. générer .oida/report.sarif
2. uploader via github/codeql-action/upload-sarif
3. vérifier que le workflow finit sans erreur
Critère

Le rapport doit inclure :

sarif_upload:
  status: pass|not_run|failed
  run_id:
  code_scanning_visible: true|false|unknown
Phase 4.6-E — provider regression baseline, mais optionnel

Phase 4.6 peut aussi préparer un baseline provider réel, mais uniquement en workflow_dispatch, jamais par défaut.

oida-code calibration-eval datasets/calibration_v1 \
  --llm-provider openai-compatible \
  --provider-profile deepseek \
  --api-key-env DEEPSEEK_API_KEY \
  --max-provider-cases 4

Règles :

- workflow_dispatch only
- secret via secrets context only
- add-mask si une valeur dynamique sensible est manipulée
- aucun provider sur PR fork
- aucun provider sur push par défaut
- métriques = contract compliance, pas benchmark public

GitHub documente ::add-mask::VALUE pour empêcher une valeur sensible d’apparaître dans les logs.

Ce bloc peut rester not_run si tu ne veux pas encore dépenser d’API.
