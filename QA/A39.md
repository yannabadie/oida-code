# QA/A39.md — Phase 5.8.1: repair case_001 pytest evidence before second soak cycle

## Verdict on the Phase 5.8 first cycle

Verdict: accepted_with_caveats.

The first operator-soak cycle is valid as a process test, not as evidence that the gateway is useful on real code yet.

What landed is meaningful:

- operator-soak.yml exists and is workflow-dispatch only.
- case_001_oida_code_self was dispatched twice.
- the first run exposed a path bug.
- the second run succeeded at the workflow level.
- cgpro produced the human label and UX scores.
- the aggregator stayed honest with cases_completed=1 and recommendation=continue_soak.

But the operator label is insufficient_fixture, because the pytest tool exited as status=error at ~193 ms and emitted no [E.tool.pytest.0] evidence. That means the first cycle proved the soak protocol and the cgpro channel, but it did not prove that the gateway-grounded verifier gave useful evidence on case_001.

The project already learned this lesson in Phase 3: a signal can look plausible while being confounded by a bad measurement path. The Phase 3 report explicitly retracted the apparent empirical success because the metric was confounded by session length and a weak outcome proxy. 

PHASE3_AUDIT_REPORT



So the next move is not to dispatch case_002 / case_003. The next move is to repair case_001’s pytest evidence path first.
