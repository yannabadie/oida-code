# QA/A42 — Phase 6.0 review

**Verdict: Accepted with conditions.**

## Verdict détaillé

Phase 6.0 est acceptée comme phase protocole + beta partielle, pas comme preuve que la beta humaine a déjà produit du signal.

Le pattern pack-first, recruitment-second est valide selon ma lecture de QA/A41 : les critères #7 et #8 autorisent explicitement un not_run documenté, #9 autorise une phase qui reste partielle, et #10 ne devient réellement testable qu'une fois des runs complétés. Le rapport Phase 6.0 documente précisément cette lecture, avec recrutement et sélection de repos séquencés après le pack.

Je ne demande donc pas de réviser QA/A41.

Je demande seulement de conserver le libellé public suivant : **Phase 6.0 accepted as protocol-established / partial beta; human validation still pending.**

### Condition 1 : ne pas présenter les scores 0.0 du zero-feedback state comme une mesure humaine.

Ils sont acceptables comme initialisation d'agrégat, mais pas comme donnée opérateur.

Dans le rapport, les lignes "setup_friction measured" et "usefulness measured" sont merge-acceptables parce qu'elles pointent vers un état zéro, mais elles peuvent être sur-lues.

Je recommanderais de reformuler en prochaine micro-doc : **"initialized in aggregate; no human measurement yet"**.

### Condition 2 : avant d'ingérer le premier retour externe, renforcer la validation du formulaire.

Le script valide bien les axes 0/1/2 et le label opérateur, mais les identifiants critiques sont convertis en chaîne vide si absents, et deux booléens critiques prennent `False` par défaut si le champ est absent.

Pour une beta ouverte, je veux que `beta_run_id`, `beta_case_id`, `beta_operator`, `target_repo`, `named_claim`, `pytest_scope`, `contract_violation_observed`, et le booléen de fuite de champ officiel soient **explicitement présents**.

### Condition 3 : garder une séparation stricte entre retours humains et lane adversarial IA.

Le plan `reports/ai_adversarial/` + `agent_label` séparé est correct.

Mais l'agrégateur cherche récursivement les fichiers dont le nom contient `beta_feedback`; si quelqu'un lance le script avec une racine trop large, il peut ingérer un fichier qui ressemble à du feedback beta.

Avant la première soumission IA adversarial, ajouter soit un champ `feedback_channel: human_beta`, soit un test qui prouve que `reports/ai_adversarial/` n'est jamais ingéré par le run beta.

### Condition 4 : ne pas considérer les 5 cas Tier-5 comme baseline externe.

Le rapport dit déjà correctement que ces cas sont de la validation interne et ne prédisent pas la distribution beta.

Il faut garder cette phrase, car elle protège contre le drift "tout est utile" quand les premiers opérateurs externes arrivent.

Les gates techniques de Phase 6.0 sont cohérentes avec ce verdict : le commit Phase 6.0 annonce les 14 livrables, +53 tests, ruff/mypy clean, zéro nouvelle dépendance, zéro runtime MCP, zéro tool-calling provider, zéro changement du core vendored, puis le commit suivant a rempli les IDs CI verts.

Le tidy postérieur est acceptable : il déplace du legacy, ajoute `docs/legacy/MOVED.md`, met à jour les références concrètes, et déclare zéro contenu Phase 6.0 touché.

**Verdict opérationnel** : tu peux committer A42 comme acceptation conditionnelle et ouvrir la fenêtre beta, à condition de traiter les conditions 2 et 3 avant le premier feedback réellement agrégé.

## Signaux à surveiller

### STOP immédiat

- **STOP** si un artefact beta sort du cadre diagnostic. Exemples concrets : `gateway_status` absent ou différent de `diagnostic_only`, rapport qui suggère une décision produit, champ officiel non nul, artefact manquant, fichier à 0 bytes, ou manifest qui ne liste pas tous les outputs attendus.
- **STOP** si le run s'exécute dans un contexte interdit par le pack beta : repo avec secrets, fork PR, cible trop large, permissions d'écriture, egress réseau non prévu, ou opérateur qui contourne le pre-dispatch local gate.
- **STOP** si le feedback n'est pas humain. Le formulaire dit explicitement que le jugement doit venir d'un lecteur humain et qu'un LLM ne doit pas remplir le formulaire.
- **STOP** si deux opérateurs indépendants reproduisent la même rupture contractuelle ou le même artefact incohérent. Un seul incident peut être un bug de cas. Deux incidents de même classe indiquent un défaut du protocole.
- **STOP** si un même bundle pinné, sur même commit et même scope, donne deux résultats incompatibles sans changement d'environnement documenté. La beta mesure l'usage humain, pas la non-déterminisme.

### PIVOT

- **PIVOT** si `setup_friction` vaut 0 ou 1 sur deux retours, et que les raisons pointent vers l'écriture du bundle.
- **PIVOT** si un seul opérateur abandonne avant le premier run à cause des 8 fichiers du bundle, des caps de longueur, des IDs d'évidence, ou de la sélection allowed_fields. Avec un n très faible, un abandon qualifié vaut plus qu'une moyenne.
- **PIVOT** si `summary_readability` vaut 0 ou 1 mais que le run est techniquement sain. Cela signifie que le problème est le rapport, pas le gateway.
- **PIVOT** si `evidence_traceability` vaut 0 ou 1 alors que l'opérateur comprend le résumé général. Cela indique que le lien claim → evidence → tool scope n'est pas assez direct.
- **PIVOT** si `actionability` vaut 0 ou 1 et que l'opérateur écrit "je comprends le résultat mais je ne sais pas quoi faire". C'est le signal le plus important pour décider si Phase 6.1 doit toucher le rapport plutôt que le bundle.
- **PIVOT** si `would_use_again` est `no` sur le premier retour et que la raison est structurelle. Exemples : "trop lourd pour une PR réelle", "je ne sais pas quel claim choisir", "je dois lire trop de JSON", "la sortie ne me donne pas de prochaine action".
- **PIVOT** si `unclear_count` ou `insufficient_fixture_count` augmente avant les vrais labels positifs/négatifs. Cela veut dire que la beta ne mesure pas encore l'utilité du diagnostic ; elle mesure l'incapacité du protocole à cadrer un cas propre.
- **PIVOT** si `false_positive_count` ou `false_negative_count` apparaît sur un cas dont le scope pytest est clair, stable, et déjà accepté par l'opérateur. Dans ce cas, ne pas corriger la doc d'abord : examiner le mapping claim/evidence/policy.
- **PIVOT** si l'opérateur passe le local gate mais échoue en GitHub-hosted run pour une raison non documentée. Cela révèle un gap de reproductibilité entre CLI locale et workflow.
- **PIVOT** si le premier retour humain demande à lire `grounded_report.json` pour comprendre le résultat principal. Le feedback form attend que Step Summary + summary.md suffisent pour la compréhension générale.

### Lecture par n

- **À 1 retour**, ne pas regarder les moyennes. Lire d'abord : le label opérateur, la raison du label, le plus mauvais axe 0/1/2, `would_use_again`, et les notes de bundle authoring.
- **À 2 retours**, regarder les répétitions. Même axe faible deux fois = signal de Phase 6.1. Deux axes faibles différents = poursuivre la beta, sauf si l'un des deux touche au cadre diagnostic.
- **À 3 retours**, seulement alors commencer à lire `operator_usefulness_rate` comme tendance faible. Avant cela, le texte libre et la classe d'échec dominent.

## Phase 6.1 préfiguration

Si la beta survit, j'attaquerais d'abord **bundle generation helper**.

Raison : QA/A41 prédisait déjà le bundle authoring comme premier vrai problème produit, et Phase 6.0 confirme dans ses docs que le bundle actuel reste manuel, multi-fichiers, et non trivial. Mais ce choix ne doit être activé que si les retours humains confirment la friction.

### Hypothèse A — choisir bundle generator

**Évidence suffisante** : `setup_friction <= 1`, opérateur capable de lire le rapport une fois le run passé, mais temps perdu sur les 8 fichiers, la sélection du claim, les IDs d'évidence, les caps de longueur, le scope pytest, ou les fichiers policy/definitions/tools.

**Autre évidence forte** : `would_use_again: yes|maybe` accompagné de "je l'utiliserais si le bundle était préparé".

Dans ce scénario, Phase 6.1 doit produire un helper minimal, pas une nouvelle architecture.
- **Entrée** : repo local, intent court, claim type, scope pytest, target ref.
- **Sortie** : squelette bundle + checklist humaine + refus si le target est hors cadre beta.

### Hypothèse B — choisir UX simplification + guided wizard

**Évidence suffisante** : le bundle n'est pas le blocage principal, mais les opérateurs se trompent entre local CLI, workflow dispatch, composite Action, artefacts à lire, ou ordre de lecture.

**Signal typique** : `setup_friction` bas ou moyen, mais commentaires "je ne savais pas quelle option lancer" ou "je ne savais pas quand remplir le formulaire".

Dans ce scénario, ne pas générer plus de fichiers. Faire plutôt un assistant CLI guidé qui pose les questions dans l'ordre, imprime la prochaine commande, puis rappelle quel artefact lire.

### Hypothèse C — choisir report structure redesign

**Évidence suffisante** : run complété, bundle acceptable, mais `summary_readability`, `evidence_traceability`, ou `actionability` faibles.

**Signal typique** : opérateur sait lancer le run mais ne sait pas relier la claim au scope, ne sait pas interpréter `verification_candidate`, ou ne sait pas quelle prochaine action humaine prendre.

Dans ce scénario, modifier d'abord `summary.md` et Step Summary. Ne pas toucher au cœur du gateway tant que le problème est la lecture.

### Décision avec n retours

- **1-2 retours** : privilégier le pire échec répété, pas le plus gros chantier. Si un opérateur échoue au bundle et un autre échoue au rapport, continuer la beta avec un troisième cas avant de trancher.
- **3-5 retours** : choisir la branche dont l'échec est le plus fréquent et le plus précoce dans le parcours. Échec avant run = bundle ou UX launch. Échec après artefacts = report redesign. Échec sur label faux positif/négatif = policy/prompt/fixture, pas une des trois hypothèses produit.

La lane adversarial IA peut informer les fixtures et les garde-fous, mais elle ne doit pas choisir Phase 6.1 à la place des opérateurs humains.

## Pièges identifiés

1. **Le mot "completed" peut dériver.** Un run complet doit signifier : local gate passé, GitHub-hosted run terminé si cette option est utilisée, artefacts présents, `summary.md` lu, `grounded_report.json` contrôlé, formulaire humain rempli. Sinon, un "run lancé" peut être compté comme "run complété".
2. **La table des gates peut être lue trop généreusement.** Dire que `setup_friction` et `usefulness` sont "measured" avec zéro feedback est techniquement défendable comme initialisation, mais humainement ambigu. Je mettrais "initialized; no human sample yet".
3. **La validation du formulaire est presque assez stricte, mais pas complètement.** Les scores et labels sont contrôlés ; les identifiants vides et booléens implicites restent le trou le plus concret avant ouverture.
4. **Le scan défensif des termes interdits ne couvre pas nécessairement tout le texte libre.** Dans le script visible, le recompte défensif regarde surtout claim, repo et scope, pas toutes les raisons écrites par l'opérateur. Il ne faut donc pas traiter ce compteur comme un scanner exhaustif des commentaires libres.
5. **Risque de pollution entre `operator_label` et `agent_label`.** Le plan séparé est bon, mais il faut le verrouiller par chemin, schéma, et tests. Une seule ingestion croisée rendrait l'agrégat humain suspect.
6. **Les 5 cas Tier-5 peuvent créer un biais d'attente.** Le rapport Phase 6.0 dit bien que leur distribution ne prédit rien pour la beta externe. À garder en tête quand le premier vrai opérateur renvoie un label moins favorable.
7. **Le doc-guard par negation detector peut être à la fois trop permissif et trop strict.** Trop permissif : une mention dangereuse proche d'un contexte négatif peut passer si elle tombe dans la fenêtre de négation. Trop strict : une phrase pédagogique légitime peut casser si elle n'utilise pas une des formulations prévues. Je n'en fais pas un blocker, mais j'ajouterais deux tests adversariaux sur docs beta avant Phase 6.1.
8. **La promesse "10-minute read" peut devenir un irritant.** Si le premier opérateur passe 45 minutes avant de comprendre le pack, ce n'est pas seulement `setup_friction`; c'est un problème de promesse documentaire. Mesurer "time to first correct local gate" dans les notes de cas, même si ce n'est pas une métrique agrégée officielle.
9. **Le template de cas peut encourager des cas trop propres.** Il faut accepter des `not_run` de target inadapté, mais ne pas laisser les opérateurs ne choisir que des repos jouets. Le bon cas beta est petit, mais pas trivial.
10. **`verification_candidate` reste un terme que les humains peuvent sur-lire.** Même si les docs l'encadrent, demander explicitement dans le feedback : "as-tu été tenté de le lire comme une décision finale ?". S'il y a une réponse oui, report redesign passe devant bundle generator.
11. **Le tidy post-Phase-6.0 est sain, mais il peut casser des liens historiques non testés.** Le mapping `docs/legacy/MOVED.md` réduit le risque, mais pendant la beta il faut surveiller les liens que les opérateurs suivent réellement.
12. **Le premier feedback peut être trop positif.** Un formulaire plein de `2` sans raisons exploitables ne suffit pas à déclencher Phase 6.1. Exiger au moins une friction concrète, une chose confuse, ou une décision humaine prise grâce au rapport.

## Conclusion

Phase 6.0 est acceptée comme protocole de beta contrôlée, avec phase humaine encore ouverte. Le pattern partial-completion est valide ; les corrections demandées portent sur l'ingestion du premier feedback, la séparation humain/IA, et la lisibilité des seuils STOP/PIVOT.
