E3 — objectif

E3 doit répondre à cette question :

Quels estimateurs peuvent remplir capability, benefit, observability, tests_pass, operator_accept, completion et edge_confidence sans mentir sur leur source, leur confiance, ni leur caractère officiel ?

E3 ne doit pas encore répondre à :

Quel est le vrai V_net du code ?

La sortie attendue d’E3 est un contrat d’estimation, pas une fusion officielle.

Modèle central
EstimateField = Literal[
    "capability",
    "benefit",
    "observability",
    "completion",
    "tests_pass",
    "operator_accept",
    "edge_confidence",
]
EstimateSource = Literal[
    "tool",
    "static_analysis",
    "test_result",
    "llm",
    "hybrid",
    "heuristic",
    "missing",
    "default",
]
class SignalEstimate(BaseModel):
    model_config = ConfigDict(extra="forbid", frozen=True)

    field: EstimateField
    event_id: str | None = None

    value: float = Field(ge=0.0, le=1.0)
    confidence: float = Field(ge=0.0, le=1.0)

    source: EstimateSource
    method_id: str
    method_version: str

    evidence_refs: tuple[str, ...] = ()
    warnings: tuple[str, ...] = ()
    blockers: tuple[str, ...] = ()

    is_default: bool = False
    is_authoritative: bool = False

Règles :

source="default" => confidence = 0.0 and blocks official fusion
source="missing" => no value may be treated as real signal
source="llm" => never authoritative alone
is_authoritative=True only allowed for tool-grounded deterministic fields
Rapport d’estimation
class EstimatorReport(BaseModel):
    model_config = ConfigDict(extra="forbid", frozen=True)

    status: Literal[
        "blocked",
        "diagnostic_only",
        "shadow_ready",
        "official_ready_candidate",
    ]

    estimates: tuple[SignalEstimate, ...]
    blockers: tuple[str, ...]
    warnings: tuple[str, ...]
    recommendation: str

À ce stade, official_ready_candidate doit rester inatteignable sauf fixture synthétique très contrôlée. La vraie sortie réaliste doit être :

diagnostic_only

ou :

shadow_ready

Pas official_ready.

E3.2 — contrats par champ
1. capability

Définition :

Does the implementation possess the mechanisms needed to satisfy the intent?

Sources possibles :

static structure
tests
contract/precondition coverage
LLM analysis of intent vs implementation

Mais :

LLM-only capability estimate must not unlock official fusion.

Contrat minimal :

value:
  0.0 = implementation lacks necessary mechanism
  0.5 = unknown / unclear
  1.0 = mechanisms clearly present and tool-supported

confidence:
  depends on evidence quality, not model certainty wording

Tests :

test_capability_missing_intent_is_blocked
test_capability_llm_only_is_diagnostic_not_authoritative
test_capability_tool_supported_contract_has_higher_confidence
test_capability_default_blocks_official_fusion
2. benefit

Définition :

If the implementation works, how valuable/relevant is it to the stated intent?

Règle critique :

No intent => benefit missing.

Ne jamais déduire un bénéfice élevé depuis la complexité du code. Un code très complexe peut être hors-sujet.

Contrat minimal :

benefit source requires:
  ticket
  prompt
  issue
  PR description
  user intent

Tests :

test_benefit_missing_without_intent
test_benefit_not_inferred_from_code_complexity
test_benefit_requires_intent_reference
test_benefit_llm_only_confidence_capped

Cap recommandé :

source="llm" without tool corroboration => confidence <= 0.6
3. observability

Définition :

Can failures be detected, localized, and surfaced?

Sources déterministes possibles :

tests exist
negative-path tests
logging / metrics / traces
explicit error handling
health checks
alerts or user-facing error paths

Contrat minimal :

observability = weighted signal from:
  test coverage proxy
  negative path coverage
  logging / metrics
  error surfacing

Attention :

presence of logging != useful observability
presence of test file != failure observability

E3.3 — LLM estimator interface contract

E3 peut préparer le LLM, mais pas encore l’utiliser comme vérité.

Créer seulement le contrat d’appel :

class LLMEstimatorInput(BaseModel):
    intent: str
    event: NormalizedEvent
    evidence_view: EventEvidenceView
    neighboring_events: tuple[NormalizedEvent, ...] = ()
    allowed_fields: tuple[EstimateField, ...]
class LLMEstimatorOutput(BaseModel):
    estimates: tuple[SignalEstimate, ...]
    cited_evidence_refs: tuple[str, ...]
    unsupported_claims: tuple[str, ...]

Règles :

LLM output must cite evidence_refs.
Unsupported claims are warnings.
LLM cannot set is_authoritative=True.
LLM cannot emit official V_net.
LLM cannot override tool failures.
LLM confidence is capped unless corroborated.

Cap de confiance recommandé :

llm-only estimate:
  confidence <= 0.6

llm + deterministic evidence:
  confidence <= 0.8

tool-grounded deterministic:
  confidence may reach 1.0 for narrow fields only

Pourquoi : le projet entier repose sur “vérifier plutôt que croire”. C’était déjà la direction du brainstorming AgentV-style : le LLM peut planifier et expliquer, mais la vérité vient des outils et des contre-exemples.
