Mais avant Block B, je veux un micro-patch A2.5 très ciblé : aligner la sémantique de U(t) avec la sémantique de visited.

Le risque actuel :

_collect_visited() only counts:
  read, grep, tool_call

is_progress_event() counts:
  any action whose scope intersects U(t)

Donc une action edit/write sur un fichier non observé peut être considérée comme un progrès sans que le fichier sorte de U(t). Ça peut créer un faux progrès répété.

Je veux que tu choisisses explicitement une des deux sémantiques et que tu l’inscrives dans le code + tests.

Option recommandée :

U(t) = unobserved audit resources
visited(t) = resources observed through read/grep/glob/tool_call
progress_event = observation of a new in-surface resource OR newly closed obligation
edit/write = not exploration progress by itself
edit/write may be paper_gain only if it first-touches a pending-obligation resource

Autrement dit, un Edit peut être utile, mais il ne doit pas fermer U(t) juste parce qu’il a un scope. Il devient un paper_gain s’il touche pour la première fois une obligation pendante dans le segment ; il devient un vrai progress_event seulement s’il ferme une obligation ou si un verifier/evidence delta prouve un progrès.
