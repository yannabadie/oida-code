Phase 4.2 — objectif

Phase 4.2 répond à une seule question :

Le verifier forward/backward peut-il demander des outils autorisés, recevoir leurs résultats sous forme d’evidence refs, puis faire valider ses claims par l’agrégateur existant ?

Elle ne répond pas encore à :

- le code est safe
- le code est merge-ready
- le code est correct
- V_net est officiel
- corrupt_success est officiel

AgentV-RL justifie bien cette direction : le papier formalise un verifier agentique avec agents forward/backward, raisonnement multi-tour et usage d’outils pour ancrer la validation au lieu de rester dans du jugement verbal.
Mais OIDA-code doit rester plus strict : les outils produisent des preuves, le LLM propose des claims, et l’agrégateur garde le dernier mot.

Règles de sécurité non négociables

Les recommandations OWASP sont directement applicables ici : donner aux agents le minimum d’outils nécessaires, appliquer des permissions par outil, séparer les niveaux de confiance, et exiger une autorisation explicite pour les actions sensibles.

Donc Phase 4.2 doit être :

read-only by default
allowlist only
no shell passthrough
no destructive command
no network by default
no env var exposure
no secret in prompt
no tool execution unless explicitly requested

OWASP signale aussi les attaques agentiques spécifiques : thought/observation injection, manipulation d’outils et poisoning de contexte.
Les sorties d’outils doivent donc être traitées comme des données non fiables, pas comme des instructions.

Si tu utilises MCP plus tard, attention : OWASP décrit MCP comme une nouvelle surface d’attaque où le LLM voit les descriptions d’outils et peut décider dynamiquement quoi invoquer ; les risques incluent tool poisoning, confused deputy, data exfiltration, permissions excessives, sandbox escape, et rug-pull de définitions d’outils.
Pour Phase 4.2, je recommande pas de MCP encore. Fais une registry locale simple.

Phase 4.2-A — Tool registry

Créer :

src/oida_code/verifier/tools/
  __init__.py
  contracts.py
  registry.py
  adapters.py
  sandbox.py

Contrat minimal :

ToolName = Literal[
    "ruff",
    "mypy",
    "pytest",
    "semgrep",
    "codeql",
]
class VerifierToolRequest(BaseModel):
    tool: ToolName
    purpose: str
    scope: tuple[str, ...]
    max_runtime_s: int
    max_output_chars: int
    requested_by_claim_id: str | None = None
class VerifierToolResult(BaseModel):
    tool: ToolName
    status: Literal["ok", "failed", "error", "timeout", "tool_missing", "blocked"]
    evidence_items: tuple[EvidenceItem, ...]
    findings: tuple[Finding, ...] = ()
    warnings: tuple[str, ...] = ()
    blockers: tuple[str, ...] = ()
    runtime_ms: int

Chaque outil doit être un adapter déterministe :

tool request
→ validation
→ execution constrained
→ parsed deterministic evidence
→ EvidenceItem refs

Pas de texte brut d’outil directement réinjecté au LLM sans filtrage.

Phase 4.2-B — Tool policy

Créer :

class ToolPolicy(BaseModel):
    allowed_tools: tuple[ToolName, ...]
    repo_root: Path
    allowed_paths: tuple[str, ...]
    deny_patterns: tuple[str, ...] = (
        ".env",
        "*.key",
        "*.pem",
        "*secret*",
        ".git/config",
    )
    allow_network: bool = False
    allow_write: bool = False
    max_tool_calls: int = 5
    max_total_runtime_s: int = 60
    max_output_chars_per_tool: int = 8000

Règles :

1. scope doit rester sous repo_root
2. path traversal interdit
3. fichiers secrets interdits
4. write interdit
5. réseau interdit
6. timeout obligatoire
7. output tronqué + hashé
8. tool_missing n'est pas une failure du code
9. tool_error devient uncertainty, pas preuve négative

        self,

Le moteur ne doit jamais exécuter une commande libre.

Interdit :

Autorisé :

Chaque adapter doit construire son argv lui-même.

Phase 4.2-D — Tool planner contract

Le verifier ne doit pas exécuter lui-même les outils.

Flux recommandé :

forward/backward verifier
→ proposes VerifierToolCallSpec
→ policy validates
→ execution engine runs
→ results become EvidenceItem
→ forward/backward verifier replays with enriched packet
→ aggregator decides

Donc :

LLM proposes tool specs
policy decides if allowed
adapter executes
parser creates evidence
LLM never sees raw command authority

Tests :

test_llm_proposed_tool_spec_is_validated_before_execution
test_disallowed_tool_spec_is_blocked
test_tool_result_becomes_citable_evidence
test_second_pass_claim_must_cite_tool_evidence
Phase 4.2-E — Two-pass loop seulement

Pas encore de boucle infinie multi-tour.

Phase 4.2 doit être strictement :

Pass 0:
  packet initial

Pass 1:
  forward/backward propose claims + optional tool requests

Tool phase:
  execute allowed requests

Pass 2:
  forward/backward verify again with tool evidence

Aggregate:
  accept/reject/unsupported

Budget :

max_passes = 2
max_tool_calls = 5
max_runtime_s = 60
max_provider_calls = 4

Pas de retry non borné. OWASP liste le denial-of-wallet et les boucles autonomes comme des risques des agents ; il faut donc des budgets stricts dès la première boucle outillée.

Phase 4.2-G — CLI

Commande séparée, toujours pas dans audit :

oida-code verify-grounded \
  packet.json \
  --forward-replay pass1_forward.json \
  --backward-replay pass1_backward.json \
  --tool-policy tool_policy.json \
  --forward-replay-2 pass2_forward.json \
  --backward-replay-2 pass2_backward.json \
  --out report.json

Ou, plus simple au début :

oida-code run-tools \
  tool_requests.json \
  --policy tool_policy.json \
  --out tool_results.json

oida-code verify-claims \
  packet_with_tool_results.json \
  --forward-replay ...
  --backward-replay ...

Je préfère commencer avec deux commandes :

run-tools
verify-claims

Puis seulement après :

verify-grounded

Pourquoi : ça rend la chaîne inspectable et évite de cacher l’exécution d’outils dans une commande opaque.

ADR-27 attendu
ADR-27 — Bounded tool-grounded verifier loop

Decision:
  Phase 4.2 introduces read-only, policy-gated tool execution for verifier evidence.
  Tool outputs become citable EvidenceItems and are re-checked by the existing
  forward/backward aggregator.

Accepted:
  - allowlisted tools only
  - no shell passthrough
  - read-only execution
  - per-tool timeout and output cap
  - evidence refs generated from parsed tool output
  - two-pass loop max
  - deterministic tool evidence wins over LLM claims

Rejected:
  - autonomous unbounded agent loop
  - destructive tool calls
  - network by default
  - MCP integration in Phase 4.2
  - raw tool output trusted as instruction
  - official V_net/debt/corrupt_success emission
