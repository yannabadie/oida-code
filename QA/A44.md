# QA/A44 — Resource allocation: PAT_GITHUB + HF + multi-provider in Phase 6.1'

**Verdict: Option Y — Phase 6.1' reste centrée sur le bundle-authoring helper, avec ajout d'un corpus adversarial borné 20–50 cas ; PAT_GITHUB / HF / providers restent manual-only et n'entrent jamais dans le runtime verifier.**

## Verdict détaillé

Le nouveau contexte change la **capacité d'échantillonnage**, pas la **frontière de confiance**.

PAT_GITHUB, HF_TOKEN, et les providers ouvrent une lane de collecte, critique, et corpus. Ils ne changent pas la nature du verifier runtime. Le verifier runtime reste local, déterministe, sans egress, sans MCP runtime, sans provider tool-calling, sans write tools.

**Le script standalone peut être riche. Le runtime verifier doit rester pauvre.** C'est la bonne asymétrie.

Les commits récents vont dans le bon sens : `49cd3fb` formalise les trois lanes external-human / AI-tier / Yann-solo, interdit la contamination croisée, et transforme Phase 6.1 en Phase 6.1' sous preuve dégradée. Le commit `fd3833a` traite C2-C5 avant Phase 6.1'a : cadrage diagnostic de `verification_candidate`, suppression de l'estimation non mesurée, walkthrough du verifier loop, et table `allowed_fields`.

`docs/project_status.md` expose maintenant les trois lanes avec chemins et discriminateurs séparés, et marque l'external-human beta comme `not_run, unavailable operators`.

Le gap **G-3** reste réel : le backlog dit explicitement qu'il n'existe ni benchmark large de PRs réelles ni dataset public de validation prédictive. Mais G-3 ne doit pas avaler Phase 6.1'. La priorité immédiate reste la friction C1. La nouvelle allocation de ressources doit servir C1, pas détourner vers un benchmark complet prématuré.

Phase 6.1' doit livrer :
- un bundle schema explainer
- un exemple non-self-audit
- un générateur minimal de squelette bundle
- un corpus adversarial borné de 20–50 cas publics Python pour stress-test
- une lane de collecte manuelle strictement séparée

Ce corpus ne ferme pas G-3. Il devient seulement un **seed corpus** pour réduire l'auto-référence des exemples. G-3 ne sort du backlog qu'après ADR séparé, méthodologie séparée, politique de labels séparée, et rapport séparé.

## Frontière manual-vs-runtime

La frontière tient pour PAT_GITHUB et HF_TOKEN si elle est écrite comme une **règle structurelle**.

### Les 12 règles

1. Les scripts avec egress vivent dans `scripts/` ou `tools/manual/`, jamais dans `src/oida_code/`.
2. Ils sont invoqués manuellement, jamais par défaut en CI.
3. Ils exigent une variable d'environnement explicite.
4. Ils refusent de tourner sans flag d'intention, par exemple `--manual-egress-ok`.
5. Ils produisent des artefacts sous `reports/`, `datasets/`, ou `calibration/`, jamais dans le runtime path.
6. Ils n'ajoutent aucune dépendance runtime.
7. Ils n'importent pas le verifier runtime dans un mode qui déclenche réseau.
8. Ils ne modifient jamais un repo cible.
9. Ils ne poussent jamais de branche.
10. Ils ne créent jamais de PR.
11. Ils n'appellent jamais une API provider depuis `verify-grounded`.
12. Ils n'appellent jamais GitHub ou HF depuis `verify-grounded`.

### Ce que PAT_GITHUB peut / ne doit pas faire

**Peut servir à :** lister repos publics ; lister PRs ; lister commits d'une PR ; lister fichiers touchés ; cloner en read-only ; récupérer metadata publique ; construire un manifest de candidats.

**Ne doit pas servir à :** merger ; commenter ; reviewer ; pousser ; modifier issue/PR ; ouvrir des checks ; écrire dans les repos cibles ; accéder à des repos privés sans ADR spécifique.

### Ce que HF_TOKEN peut / ne doit pas faire

**Peut servir à :** télécharger un dataset public ou gated explicitement accepté ; télécharger un modèle open-weight pour évaluation locale optionnelle ; uploader un dataset seulement après politique dédiée.

**Ne doit pas servir à :** publier automatiquement ; uploader du code privé ; uploader des diffs sans analyse de licence ; synchroniser un corpus brut par défaut ; transformer une lane de collecte en produit.

### Ce que les provider API keys peuvent / ne doivent pas faire

**Peuvent servir à :** AI-tier cold-reader critique ; proposition de cas candidats ; critique du bundle generator ; comparaison qualitative entre lecteurs.

**Ne doivent pas servir à :** remplir `human_beta` ; produire `operator_label` ; décider un label de vérité ; écrire dans l'agrégat humain ; exécuter le verifier runtime ; traiter du code privé ou sensible.

### Règles de test recommandées

- Aucun fichier sous `src/oida_code/` ne doit contenir `PAT_GITHUB`, `HF_TOKEN`, ou les noms de variables provider.
- Aucun module du verifier ne doit importer `requests`, `httpx`, `huggingface_hub`, ou un client provider.
- Les scripts egress doivent porter un marqueur visible, par exemple `MANUAL_EGRESS_SCRIPT = True`.
- Tout script egress doit avoir `--dry-run` par défaut.
- Toute sortie de collecte doit inclure `source_url`, `repo`, `commit_sha`, `pr_number`, `license_observed`, `public_only`, et `collected_at`.

### La frontière à écrire dans ADR-53

> **Manual data acquisition may use network credentials. The verifier runtime may not. Manual scripts can collect candidate evidence; they cannot produce human feedback, cannot produce runtime decisions, and cannot relax structural pins.**

## Calibration dataset boundaries

Oui, PAT_GITHUB permet de construire un dataset de calibration à grande échelle sans violer QA/A41 line 350. **Mais seulement si le dataset est défini comme benchmark structurel, pas comme feedback opérateur.**

Le bon objet n'est pas : "des utilisateurs disent que l'outil est utile". Le bon objet est : "sur des PRs réelles, le pipeline sait-il construire et rejouer un cas de grounding dans un cadre fixe ?"

### Frontière acceptable

- PR publique
- repo public
- commit SHA pinné
- claim explicite ou dérivée d'un artefact public
- scope pytest ou test command documenté
- bundle généré ou préparé
- expectation structurelle documentée
- label source non-LLM ou LLM-assisté puis confirmé

### Frontière interdite

- LLM remplit le formulaire opérateur
- LLM produit `operator_label`
- LLM note `setup_friction`
- LLM note `would_use_again`
- LLM décide seul qu'une claim est vraie
- LLM décide seul qu'une PR est bonne
- LLM convertit son avis en signal humain
- LLM choisit la Phase suivante par vote

### Schéma minimal acceptable

```yaml
case_id:
repo_url:
base_sha:
head_sha:
pr_number:
claim_id:
claim_type:
claim_text:
test_scope:
expected_grounding_outcome:  # evidence_present | evidence_absent | tool_missing | scope_invalid | ambiguous | not_run
label_source:                # deterministic_tool_output | repository_metadata | yann_manual_review | ai_candidate_human_confirmed | unknown_not_for_metrics
selection_source:
llm_assist_used:
human_review_required:
```

`expected_grounding_outcome` doit être structurel. Ne pas utiliser de label produit, de label utilisateur, de label de sécurité.

`label_source` doit être strict. Valeurs interdites : `llm_only`, `agent_vote`, `provider_consensus`, `cold_reader_label`, `human_beta` si l'humain est Yann ou un agent.

### LLM acceptable / interdit

**Acceptable :** proposer des PRs candidates ; résumer un diff public pour aider Yann ; suggérer un `claim_id` ; repérer des cas adversariaux ; critiquer une fiche de cas.

**Interdit :** clore le label ; remplir un feedback humain ; produire une métrique opérateur ; devenir oracle de vérité ; choisir les cas qui seront seuls publiés sans audit.

### Naming policy

Le dataset doit être appelé `calibration_seed` ou `structural_grounding_corpus`. **Ne pas l'appeler** `human_beta`, `operator_feedback`, ou `validation_dataset` avant ADR méthodologie.

### Première version

20–50 cas suffisent. Au-delà, le risque est de produire du volume avant d'avoir stabilisé la taxonomie.

Le corpus doit couvrir les 7 `claim_type` existants, mais sans forcer l'égalité parfaite. Inclure des **cas négatifs** : test scope absent ; package impossible à installer ; PR trop large ; claim trop vague ; diff sans test ; repo archivé ; changement hors Python ; fichier supprimé ; dependency failure ; flaky test suspect.

Chaque cas doit avoir une **raison d'inclusion**. Chaque exclusion doit avoir une **raison d'exclusion**. Les exclusions sont aussi du signal.

## HuggingFace usage policy

### (a) Télécharger un dataset existant de PRs / commits

**ACCEPT read-only survey, DEFER phase gate usage.**

CodeXGLUE et CommitPackFT peuvent aider à comprendre la distribution des messages de commit, trouver des patterns de claims, tester extraction de changements, enrichir une taxonomie de cas. Ils ne doivent pas décider Phase 6.1', ne doivent pas être la base unique du corpus, et ne doivent pas être traités comme PR feedback.

Avant usage dans un rapport : pinner dataset id, pinner revision, lire la dataset card, noter licence, noter limites, noter si les données sont commits / PRs / tâches synthétiques, documenter le mismatch avec oida-code.

### (b) Upload du dataset calibration généré vers HF Hub

**DEFER public upload until ADR + data card + license pass.**

Publier un corpus extrait de PRs réelles crée un risque licence / attribution / vie privée / secret / Terms.

Première étape autorisée : publier un **manifest local** ; ne stocker que URLs publiques, SHAs, PR numbers, claim ids, et outcomes structurels ; ne pas stocker raw source ; ne pas stocker raw diff ; ne pas stocker provider transcript contenant du code ; ne pas stocker données de repos privés.

Upload acceptable plus tard si : dataset card complète ; licence claire ; source repos publics uniquement ; attribution par repo ; opt-out process ; secret scan ; PII minimization ; manifest reproductible ; aucun contenu non-public ; aucune donnée issue d'un PAT privé.

Format public préféré : **manifest-only**, regenerated-on-demand, raw code fetched by user from original repo, no vendored source blobs.

### (c) Télécharger des modèles open-weight pour evaluator local optionnel

**ACCEPT optional local evaluator lane, DEFER implementation unless needed by corpus triage.**

Si implémenté : modèle pinné ; licence notée ; poids non vendored ; cache local ignoré par git ; sortie marquée `ai_tier` ; aucun label humain ; aucun effet sur `reports/beta/`.

## Multi-provider panel sizing

Un panel 8–10 agents **ne produit pas automatiquement un meilleur signal**. Il produit plus de texte, plus de coûts, plus d'exposition de données, plus de risque Goodhart.

Le gain marginal après 3–4 cold readers indépendants est **faible** si le prompt et les inputs sont identiques.

La diversité utile n'est pas seulement le nombre de modèles : provider différent, famille de modèle différente, **rôle différent**, prompt différent, surface différente, moment différent.

### Panel recommandé par cycle

- 3 cold readers généralistes
- 1 adversarial protocol reviewer
- 1 implementer persona si le sujet est generator
- pas plus de 5 sorties à agréger

Panel 8–10 acceptable seulement pour : régression majeure ; rapport final de corpus ; décision de retirer un finding ; divergence persistante ; audit de wording public.

**Même dans ce cas, ne pas voter.** Ne pas dire "8 agents sur 10 donc vrai". Dire : convergence qualitative, divergence qualitative, suggestions rejetées, suggestions actionnées, risques nouveaux.

### Bonne métrique AI-tier

Pas `n_agents_positive`. Préférer :

- `finding_reproduced_after_patch`
- `finding_resolved_by_patch`
- `new_finding_introduced`
- `contract_violation_suggested`
- `human_review_needed`

### Limiter le contexte envoyé aux providers

- Public docs : OK
- Public PR diffs : possible, mais minimiser
- Private repo : non
- Secrets : non
- Large raw source : non, sauf nécessité et public-only

Pour les bundles, envoyer : manifest, schema, summary, excerpts minimaux, pas tout le repo.

## Phase 6.1' option choice

**Recommandation : Y — Bundle generator + adversarial corpus.**

### Pourquoi pas X (bundle generator only)

X protège le focus, mais il laisse le generator être testé seulement sur des exemples auto-référentiels. Or C1 dit précisément que le bundle est trop difficile à dériver depuis les exemples actuels. Un helper sans corpus réel risque de résoudre le mauvais problème.

### Pourquoi pas Z (pivot G-3 complet)

Z attaque G-3 directement, mais trop tôt. Un dataset de calibration complet exige : sampling policy, license policy, label policy, publication policy, methodology ADR, negative cases, reproducibility story, evaluator story, report format. Ce serait plus gros que Phase 6.1'. Z remettrait C1 sous le tapis. Z obligerait Yann à produire manuellement des bundles avant même de livrer le helper qui réduit cette friction.

### Y est le compromis

Phase 6.1' devient :

> Minimal bundle-authoring helper, tested against a bounded adversarial corpus of real public Python PR cases, with all data acquisition manual-only and all labels separated from human-beta.

### Sous-blocs recommandés

- **Phase 6.1'a-pre** — data lane contract.
- **Phase 6.1'a** — bundle schema explainer + one non-self-audit worked example.
- **Phase 6.1'b** — minimal bundle skeleton generator.
- **Phase 6.1'c** — 20–50 public Python PR seed corpus.
- **Phase 6.1'd** — generator stress-test on seed corpus.
- **Phase 6.1'e** — AI-tier re-run + Yann-solo dogfood.

Le corpus ne doit pas attendre la fin du generator. Il faut au moins 5–10 cas candidats avant de designer le generator. Mais il ne faut pas construire 500 cas avant le generator.

### Cible initiale

- 10 repos publics Python
- 20 PRs candidates
- 7 claim types couverts au moins une fois si naturellement possible
- 5 cas négatifs
- 1 worked example complet
- 1 bundle skeleton par cas candidat

### Le générateur minimal NE DOIT PAS

- auto-labeler
- produire `operator_label`
- produire un avis utilisateur
- décider du code
- faire des `provider call`
- faire un `GitHub write`
- faire un `HF upload`
- commenter une PR
- commenter une issue

### Critères de succès Phase 6.1'

- generator crée un squelette bundle valide pour au moins 10 cas publics
- 0 cas écrit dans les repos cibles
- 0 provider call dans generator
- 0 HF call dans generator
- 0 GitHub write permission utilisée
- 0 contamination `reports/beta/`
- chaque cas a provenance pinnée
- chaque cas a reason d'inclusion
- chaque exclusion a reason d'exclusion
- Yann-solo note le temps et les edits nécessaires
- AI-tier ne retrouve pas C1 sous la même forme

## Nouveaux pièges 27+

27. **Confondre capacité de collecte et maturité méthodologique.** Avoir PAT_GITHUB ne crée pas un benchmark.
28. **Corpus auto-sélectionné.** Si Yann choisit les repos qu'il aime, le corpus peut devenir trop propre. Documenter la stratégie d'échantillonnage.
29. **Survivorship bias des PRs mergeées.** Les PRs mergeées ont souvent déjà passé un processus humain. Elles ne représentent pas les claims faibles.
30. **Popularité comme proxy de qualité.** Les gros repos Python sont mieux testés, mieux documentés, et moins représentatifs des repos ordinaires.
31. **Corpus trop Python-package.** Varier CLI, libs, web, data, test tooling.
32. **PR trop grandes.** Les grosses PRs produisent de mauvais cas beta. Limiter fichiers touchés, lignes diffées, surface testable.
33. **PR trop petites.** Les micro-PRs triviales peuvent flatter le generator. Cas simples, pas jouets.
34. **Claim implicite inventée.** Une claim doit être reconstructible depuis PR title / description / commit / tests / diff.
35. **Labels LLM déguisés en labels structurels.** `ai_candidate_human_confirmed` doit signifier confirmation réelle.
36. **Provider data leakage.** Envoyer des diffs publics à des providers est moins risqué que du privé, mais c'est quand même une exposition. Réduire les payloads.
37. **Fuite future vers training data.** Outputs enrichis, labels, summaries peuvent créer un dataset nouveau. Politique nécessaire.
38. **Secret déjà présent dans repo public.** Scanner avant provider calls et avant publication.
39. **Licence ignorée.** Public ne veut pas dire librement republiable. Manifest-only au début.
40. **HF upload trop tôt.** Public = cloné, indexé, réutilisé. Coût d'une correction tardive élevé.
41. **Drift de dependency.** Ajouter `huggingface_hub`, `datasets`, ou clients GitHub comme dépendances runtime serait une régression. Extra dev/manual si nécessaire.
42. **Rate-limit driven sampling.** Logguer les skips rate-limit.
43. **Token overprivileged.** Refuser si le PAT a write permissions, si possible.
44. **CI accidentelle.** Test ou grep pour empêcher qu'un script manual-only finisse en workflow.
45. **Local clone pollution.** Utiliser temp dirs et `.gitignore`.
46. **Benchmark qui optimise le generator.** Garder des holdout cases.
47. **AI panel comme majority vote.** Toujours agréger par finding, pas par vote.
48. **Prompt reuse.** Varier les rôles plus que le nombre.
49. **Confusion corpus vs feedback.** Un cas de corpus n'est pas un opérateur. Un résultat de stress-test n'est pas un score d'usage.
50. **Yann-solo surcharge.** Avec beaucoup de données, Yann peut devenir le goulot. Limiter N volontairement.
51. **Publish pressure.** Ne publier que ce que la méthodologie permet de défendre.
52. **Private-access ambiguity.** Public-only par défaut. Refuser tout repo privé sans ADR séparé.
53. **Provider outputs contaminent dataset.** `reports/ai_adversarial/` séparé de `reports/calibration_seed/`.
54. **Corpus sans negative cases.** Inclure `not_run`, `scope_invalid`, `ambiguous`.
55. **Métriques trop séduisantes.** Préférer count par cause d'échec et reason logs.
56. **Oublier le but initial.** Phase 6.1' = réduire la friction bundle. Le corpus est au service de cela.

## Action immédiate

**Sous-bloc prioritaire à shipper : Phase 6.1'a-pre — manual data lane contract + seed corpus indexer dry-run.**

Ne pas commencer par 50 cas. Ne pas commencer par HF upload. Ne pas commencer par open-weight evaluator. Ne pas commencer par panel 10 agents. **Commencer par la frontière.**

### `build_calibration_seed_index.py` doit être dry-run par défaut

**Flags minimaux :**
- `--repo owner/name`
- `--max-prs`
- `--public-only`
- `--output reports/calibration_seed/index.json`
- `--manual-egress-ok`
- `--no-provider-calls`
- `--no-hf-upload`

**Le script doit collecter metadata seulement au début :** repo URL, PR number, title, base SHA, head SHA, changed files list, GitHub labels si publics, merge status si public, candidate reason, exclusion reason.

**Pas de raw diff dans la première version. Pas de raw source. Pas de provider call. Pas de HF call. Pas de clone obligatoire.**

### Critères de succès mesurables

- 5 à 10 PR candidates publiques collectées en metadata-only
- 5 exclusions documentées avec reason
- tous les records ont repo URL, PR number, base/head SHA
- aucune sortie dans `reports/beta/`
- aucun import réseau dans `src/oida_code/`
- aucun token écrit dans artefacts
- script échoue sans `--manual-egress-ok`
- script échoue si `--public-only` est absent
- ruff clean
- mypy clean
- tests lane-isolation passent

## Décision finale A44

> **Use the new resources to build a bounded, public-only, manual data acquisition lane that stress-tests bundle authoring. Do not pivot into full G-3 calibration yet. Do not let PAT_GITHUB, HF_TOKEN, or providers cross into runtime verifier or human-beta feedback.**
